# 烽火笔锋训练后端一周双人并行开发计划

## 1. 计划目标

本计划用于把当前训练后端从“研发可用、可自测”推进到“可稳定联调、可灰度交付”的 Beta 状态。

计划周期：

- `2026-03-18` 至 `2026-03-24`

计划前提：

- 本周只聚焦训练后端主链路
- 本周不启用 Chroma 进入训练主流程
- 本周不做 TTS、图片、角色系统的大改
- 本周不做前端大改，只保证 API 契约稳定
- 两位开发人员全程并行，尽量互不干涉，最后两天只做集成与收口

本周目标结果：

1. 训练主链路接口可稳定联调
2. 数据库初始化和升级流程从“补表”升级为“正式迁移”
3. 训练域继续降耦合，减少 `TrainingService` 再次膨胀
4. 训练接口具备单元测试、集成测试、数据库自检与 smoke 自检能力
5. 输出可交付的开发文档、部署文档、回归文档

---

## 2. 本周总优先级

1. 建立正式数据库迁移体系，停止依赖运行时 `create_all`

3. 补训练接口的真实数据库集成测试与回归链路
4. 固化初始化、自检、迁移、冒烟脚本

### P1

2. 继续把流程参数、推荐参数、结局参数配置化
3. 补齐文档和周交付说明

### P2

1. 鉴权
2. 限流
3. 监控告警
4. Chroma 检索增强
5. 前端深度联调

说明：

- `P2` 不进入本周主目标，只列入下周候选

---

## 3. 双人拆分原则

为了保证一周内最大并行效率，本周采用“领域线 / 平台线”分拆。

拆分原则：

1. 两位开发不修改同一批核心文件
2. 两位开发不共享同一组测试文件
3. 训练算法与流程编排由开发者 A 负责
4. 数据库迁移、接口壳层、脚本与集成验证由开发者 B 负责
5. 直到 `2026-03-23` 才允许跨边界处理集成问题

## 8. 集成规则

为了避免双人并行时互相覆盖，本周执行以下规则：

1. 本周外部训练 API 结构冻结，除非发现阻塞性 bug，不再临时扩字段
2. 开发者 A 不改 `api/schemas.py` 和 `router`
3. 开发者 B 不改 `training_service.py` 和 `backend/training/**`
4. 双方不共享同一测试文件
5. 只有 `2026-03-23` 可以跨边界修集成问题
6. 如果某个问题必须同时修改 A、B 独占边界，直接标记为“下周处理”，不在本周强行穿插

## 10. 本周主要风险

### 风险 1：数据库迁移落地不顺

说明：

- 当前系统历史上依赖 `create_all`
- 一旦 baseline migration 和现有表结构不一致，容易在本周中后期卡住

应对：

1. Day 2 必须完成 baseline migration 验证
2. Day 3 前必须确认迁移脚本是唯一推荐路径

### 风险 2：训练服务继续膨胀

说明：

- 如果开发者 A 在 service 内继续塞逻辑，本周虽然能交付，但下周耦合会迅速反弹

应对：

1. 本周只允许在 service 中做编排
2. 新增逻辑优先进入 policy、contract、output DTO

### 风险 3：双人并行时边界不清

说明：

- 如果双方都去改 schema、service、router，很容易在 Day 5 以后爆冲突

应对：

1. 严格按独占文件边界开发
2. 任何跨边界改动都延迟到集成日

### 风险 4：测试只在内存桩通过，真实数据库不过

说明：

- 当前训练域单元测试较多，但真实数据库集成覆盖仍然要继续补

应对：

1. 开发者 B 必须补 PostgreSQL 集成测试
2. Day 6 必须做真实数据库 smoke

## 11. 一周压缩后的目标定位

本周压缩计划的目标不是把整个后端做成“生产完全体”，而是把训练后端做成：

1. 架构骨架稳定
2. 训练主链路可联调
3. 数据库可迁移
4. 测试可回归
5. 文档可交接

换句话说，本周结束后的目标状态应该是：

**训练后端 Beta 可交付，而不是全站生产完全体。**
