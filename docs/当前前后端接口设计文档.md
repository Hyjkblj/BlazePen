# 当前前后端接口设计文档

> 依据当前代码实现整理，代码为事实源。本文档主要对应：
> `backend/api/routers/game.py`、`backend/api/routers/training.py`、`backend/api/schemas.py`、`backend/api/story_schemas.py`、`frontend/src/services/*.ts`、`frontend/src/types/*.ts`、`frontend/src/utils/*.ts`。

## 1. 文档目标

本文档描述 BlazePen 当前前后端接口设计的实际状态，重点回答两件事：

1. 当前前后端 HTTP 接口如何分域、如何传参与如何返回。
2. 当前关键字段名如何设计，哪些字段只能用于故事主线，哪些字段只能用于训练主线。

本文档不以历史设计稿为准，而以当前运行中的代码契约为准。

## 2. 总体接口设计原则

### 2.1 业务主线分离

- 故事主线接口域：`/api/v1/game/*`
- 训练主线接口域：`/api/v1/training/*`
- 故事初始化兼容入口：`/api/v1/characters/initialize-story`

当前设计里，故事主线和训练主线是两套不同的会话模型，不能混用：

- 故事主线会话主键是 `thread_id`
- 训练主线会话主键是 `session_id`

这是当前最重要的命名边界。

### 2.2 Wire Contract 与前端 Canonical Model 分离

后端 HTTP wire format 统一使用 `snake_case`，前端服务层和归一化层再映射为 `camelCase`。

示例：

- 后端：`thread_id`
- 前端：`threadId`

- 后端：`current_round_no`
- 前端：`currentRoundNo`

- 后端：`decision_context`
- 前端：`decisionContext`

因此，页面、flows、hooks 不应直接消费后端原始字段，而应消费 services/utils 归一化后的前端模型。

### 2.3 统一响应包裹格式

成功响应统一为：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

错误响应统一为：

```json
{
  "code": 404,
  "message": "training session not found",
  "data": null,
  "error": {
    "code": "TRAINING_SESSION_NOT_FOUND",
    "details": {},
    "traceId": "trace-xxx"
  }
}
```

关键约束：

- 业务分支判断应优先依赖 `error.code`
- `message` 用于展示与排障，不应承载业务语义
- `error.details` 是结构化扩展上下文
- `traceId` 用于链路追踪

### 2.4 契约稳定性规则

- 训练请求 DTO 开启 `extra="forbid"`，即请求不接受未知字段
- 前端服务层负责把请求参数从 `camelCase` 转成 `snake_case`
- 前端归一化层负责把响应字段从 `snake_case` 转成 `camelCase`
- 新接口优先沿用结构化字段，不新增 message 驱动逻辑

## 3. 故事主线接口设计

故事主线当前的会话事实源字段是 `thread_id`。故事流相关恢复、继续、结局查询都围绕 `thread_id` 建立。

### 3.1 接口总览

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/api/v1/game/init` | 创建故事会话 |
| `POST` | `/api/v1/game/input` | 提交一轮故事输入 |
| `POST` | `/api/v1/game/initialize-story` | 进入初始故事场景，当前 canonical 路径 |
| `POST` | `/api/v1/characters/initialize-story` | 历史兼容路径，契约与 canonical 路径相同 |
| `GET` | `/api/v1/game/sessions` | 获取某用户最近故事会话列表 |
| `GET` | `/api/v1/game/sessions/{thread_id}` | 获取故事会话快照 |
| `GET` | `/api/v1/game/sessions/{thread_id}/history` | 获取故事历史回合 |
| `GET` | `/api/v1/game/sessions/{thread_id}/ending` | 获取故事结局摘要 |
| `GET` | `/api/v1/game/check-ending/{thread_id}` | 旧版结局检查接口 |
| `POST` | `/api/v1/game/trigger-ending` | 主动触发结局 |

### 3.2 `POST /api/v1/game/init`

用途：创建故事主线会话，返回新的 `thread_id`。

请求字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `user_id` | `string` | 否 | 用户 ID，不传时后端可自动生成 |
| `game_mode` | `string` | 是 | 当前主要使用故事模式会话 |
| `character_id` | `string` | 是 | 角色 ID，故事域里当前按字符串传输 |

响应核心字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `thread_id` | `string` | 故事会话主键 |
| `user_id` | `string` | 用户 ID |
| `game_mode` | `string` | 当前游戏模式 |
| `status` | `string` | 前端会补充为初始化状态语义 |

前端 canonical 对应：

- `thread_id` -> `threadId`
- `user_id` -> `userId`
- `game_mode` -> `gameMode`

### 3.3 `POST /api/v1/game/input`

用途：提交故事回合输入，返回当前最新场景状态。

请求字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `thread_id` | `string` | 是 | 故事会话主键 |
| `user_input` | `string` | 是 | 用户输入文本；也支持 `option:<number>` 形式 |
| `user_id` | `string` | 否 | 用户 ID |
| `character_id` | `string` | 否 | 角色 ID，用于恢复路径等场景 |

响应核心字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `thread_id` | `string` | 当前故事会话 ID |
| `status` | `string` | 当前会话状态 |
| `round_no` | `number` | 当前回合号 |
| `character_dialogue` | `string` | 角色对话 |
| `player_options` | `array` | 玩家可选项 |
| `story_background` | `string` | 背景叙述 |
| `event_title` | `string` | 当前事件标题 |
| `scene` | `string` | 当前场景标识，前端归一化为 `sceneId` |
| `scene_image_url` | `string` | 场景图 |
| `composite_image_url` | `string` | 合成图 |
| `assets` | `object` | 结构化图片资源元数据 |
| `current_states` | `object` | 当前状态值 |
| `session_restored` | `boolean` | 是否发生会话恢复 |
| `need_reselect_option` | `boolean` | 恢复后是否要求重新选项 |
| `restored_from_thread_id` | `string` | 恢复来源 thread |
| `is_event_finished` | `boolean` | 当前事件是否结束 |
| `is_game_finished` | `boolean` | 故事是否完成 |
| `snapshot` | `object` | 原始快照数据 |

前端 canonical 对应：

- `thread_id` -> `threadId`
- `session_restored` -> `sessionRestored`
- `need_reselect_option` -> `needReselectOption`
- `restored_from_thread_id` -> `restoredFromThreadId`
- `scene` -> `sceneId`

说明：

- `snapshot` 是恢复辅助字段，不建议页面直接作为首要渲染源
- 前端 `normalizeStoryScenePayload` 会优先读根字段，再回退读取 `snapshot`

### 3.4 `POST /api/v1/game/initialize-story`

用途：根据已有 `thread_id` 和 `character_id` 初始化故事起始场景。

请求字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `thread_id` | `string` | 是 | 故事会话 ID |
| `character_id` | `string` | 是 | 角色 ID |
| `scene_id` | `string` | 否 | 起始大场景 ID，默认 `school` |
| `opening_event_id` | `string` | 否 | 指定开场事件 |
| `character_image_url` | `string` | 否 | 角色图片 URL |

响应契约与 `POST /api/v1/game/input` 基本一致，仍返回一份标准的故事场景 DTO。

兼容性说明：

- `/api/v1/game/initialize-story` 是当前 canonical 路径
- `/api/v1/characters/initialize-story` 是薄包装兼容路径
- 新设计、前端新逻辑、后续文档都应优先以 `/api/v1/game/initialize-story` 为主

### 5.3 训练模式归一化

训练模式当前有一条明确规则：

- 请求入参兼容：`self_paced`
- 后端/前端 canonical 值：`self-paced`

因此：

- 前端允许把用户输入 `self_paced` 归一化成 `self-paced`
- 响应消费层不应再把 `self_paced` 当作 canonical 输出

### 5.4 场景摘要字段归一化

训练场景摘要字段当前 canonical 名称是：

- `brief`

不是：

- `briefing`

前端测试已经显式约束：

- `resumableScenario.brief` 为标准字段
- `scenario.brief` 为标准字段
- `scenarioCandidates[].brief` 为标准字段

因此后端、前端新代码都不应继续扩散 `briefing` 变体。

## 6. 当前关键字段名设计

这一节是当前最关键的字段命名结论。

### 6.1 会话主键

| 字段名 | 域 | 含义 | 备注 |
| --- | --- | --- | --- |
| `thread_id` | 故事 | 故事会话主键 | 前端映射为 `threadId` |
| `session_id` | 训练 | 训练会话主键 | 前端映射为 `sessionId` |

硬性规则：

- 故事主线只使用 `thread_id`
- 训练主线只使用 `session_id`
- 两者不能互相别名化，也不能合并为通用 `sessionId`

### 6.2 用户与角色

| 字段名 | 域 | 类型 | 说明 |
| --- | --- | --- | --- |
| `user_id` | 故事/训练 | `string` | 用户主键 |
| `character_id` | 故事 | `string` | 故事域当前通过字符串传输 |
| `character_id` | 训练 | `number` | 训练 init 请求中是数值型 |

注意：

- 字段名相同不代表类型相同
- 故事域与训练域不要强行抽成同一个前端原始 request DTO

### 6.3 场景与回合

| 字段名 | 域 | 含义 |
| --- | --- | --- |
| `scene_id` | 故事请求 | 初始化故事时指定起始场景 |
| `scene` | 故事响应 | 当前故事场景标识，前端映射为 `sceneId` |
| `scenario_id` | 训练请求/历史 | 训练场景标识 |
| `round_no` | 故事/训练 | 当前回合号 |
| `current_round_no` | 训练恢复/运行时 | 当前运行时回合号 |

### 6.4 结构化状态字段

| 字段名 | 域 | 含义 |
| --- | --- | --- |
| `current_states` | 故事 | 当前剧情状态集合 |

| `progress_anchor` | 训练 | 恢复锚点与稳定进度摘要 |

### 6.5 选择、推荐与后果

### 6.6 恢复与生命周期

| 字段名 | 域 | 含义 |
| --- | --- | --- |
| `session_restored` | 故事 | 当前故事响应是否来自恢复链路 |

| --- | --- |
| `code` | HTTP 语义状态码镜像 |

当前关键错误码：

- 训练域：`TRAINING_SESSION_NOT_FOUND`、`TRAINING_SESSION_COMPLETED`、`TRAINING_SESSION_RECOVERY_STATE_CORRUPTED`、`TRAINING_ROUND_DUPLICATE`、`TRAINING_MODE_UNSUPPORTED`、`TRAINING_SCENARIO_MISMATCH`

## 7. 命名与契约红线

以下规则建议视为当前接口设计红线：

1. 不要把故事 `thread_id` 和训练 `session_id` 抽象成同一个后端字段名。
2. 不要让页面层直接兼容后端脏字段，兼容逻辑应收敛在 services/utils。
3. 不要依赖 `message` 文本做业务判断，应依赖 `error.code`。
4. 不要把训练场景摘要字段再命名成 `briefing`，当前标准名是 `brief`。
5. 不要把 `self_paced` 继续向响应层扩散，当前 canonical 名是 `self-paced`。
6. 不要把故事响应里的 `scene` 和故事请求里的 `scene_id` 误认为完全同一字段。
7. 不要把本地缓存字段名提升为服务端契约字段名，服务端事实源仍由 `thread_id` / `session_id` 驱动。

## 8. 推荐的前后端协作方式

### 8.1 前端

- 页面、flows、hooks 只使用前端 canonical 字段，例如 `threadId`、`sessionId`、`decisionContext`
- services 负责请求拼装
- utils 负责响应归一化
- 页面不要直接解析 `snake_case` 原始响应

### 8.2 后端

- router 负责协议边界与错误码
- service/query service 负责业务与恢复模型
- schema/story_schema 负责稳定 DTO
- 新增字段优先扩展 DTO，不要隐式塞到 message 或未约束字典里

## 9. 一页结论

如果只记当前接口设计最关键的字段名，请记下面这些：

- 故事主线会话：`thread_id` / 前端 `threadId`
- 训练主线会话：`session_id` / 前端 `sessionId`
- 故事场景请求：`scene_id`
- 故事场景响应：`scene`
- 训练场景主键：`scenario_id`
- 训练模式：`training_mode`，canonical 值是 `guided | self-paced | adaptive`
- 训练摘要字段：`brief`
- 训练结构化决策上下文：`decision_context`
- 训练结构化后果事件：`consequence_events`
- 错误语义字段：`error.code`
- 链路追踪字段：`error.traceId`

这份字段边界如果保持稳定，前后端职责会更清晰，恢复链路、读模型和测试基线也会更容易维持一致。
