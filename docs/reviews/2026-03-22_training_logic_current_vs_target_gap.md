## 一句话结论

当前实现不等于“玩家以记者或其他身份，在由 LLM 实时推理出的随机故事线中穿越 6 个大场景和多个小场景，并最终生成训练报告”。

当前更准确的表述是：

`固定历史主线 + 配置化分支/补救场景 + 规则优先评估 + 可选 LLM 辅助评估 + KT/S 状态更新 + 最终训练报告`

并且要明确一点：即使按目标完整模式理解，本项目也不是要做成“无限开放、完全随机、由 LLM 自由生成历史剧情”的系统，而是“历史约束下的自适应训练系统”。

## 当前已经实现的训练业务逻辑

### 后端现状

1. 训练主线是固定历史序列，不是开放随机剧情。
   `backend/training/config/training_runtime_config.json` 已明确配置 6 个核心场景默认序列 `S1-S6`，同时开启了顺序约束。

2. 当前存在少量配置化支线，不是大规模动态生成场景网。
   `backend/training/config/scenario_bank.json` 除主线外，已经有 `S2B`、`S3R` 这类失败分支或恢复分支。

3. 回合推进是策略驱动，不是 LLM 主导。
   `backend/training/round_flow_policy.py` 结合固定序列、`next_rules`、`BranchResolver`、推荐策略推进训练。

4. 玩家决策会改变后续路径，但影响方式主要是“状态与分支”，不是“自由生成下一章剧情”。
   `backend/training/consequence_engine.py` 会根据风险结果更新 `panic_triggered`、`source_exposed`、`editor_locked`、`high_risk_path` 等运行时状态，再由分支策略决定是否进入补救路径。

5. 评估是“规则优先，LLM 可选增强”。
   `backend/training/evaluator.py` 先做规则评估，只有在开关开启时才叠加 LLM 评估；当前默认配置是关闭 LLM 评估。

6. 训练报告已经进入正式业务链路。
   `backend/training/reporting_policy.py` 已能输出 `summary`、`ability_radar`、`state_radar`、`growth_curve` 等报告结构。

### 前端现状

1. 训练页、进度页、报告页、诊断页都已经存在。
   当前仓库里已有 `Training.tsx`、`TrainingProgress.tsx`、`TrainingReport.tsx`、`TrainingDiagnostics.tsx`。

2. 前端已经能消费训练主链路，但还不能据此宣称“训练前端完整收口”。
   目前更准确的状态是：训练前端已经从“仅后端可跑”走到了“页面链路可用、可继续开发训练产品层”的阶段。

## 当前还没有达到的“完整模式”

### 前端差距

1. 训练页主流程还没有完全收口为单一事实源。
   现状里训练主 flow 仍承担较多恢复、视图投影和页面编排逻辑，说明前端还没有完全完成 `pages / flows / hooks / services / storage / contexts` 的边界稳定化。

2. 页面层的 DTO 消费边界还不够干净。
   训练页仍有兼容后端别名字段的痕迹，这意味着“页面只消费规范契约字段”这条线还没有彻底收口。

3. 报告页虽然已经落地，但读模型消费的一致性还在打磨阶段。
   也就是说，现在已经有报告页，不再是“没有 TrainingReport.tsx”，但它离“完整产品级训练洞察页”还有稳定化工作。

4. 前端还没有真正形成“多身份训练”的产品形态。
   即便存在 `identity` 输入，当前 UI 和训练流仍明显围绕“记者训练”组织，没有独立角色差异化流程。

### 后端差距

1. 当前不是 LLM 叙事引擎，而是训练策略引擎。
   如果把目标理解成“LLM 自由生成随机故事线”，那当前后端并没有实现，而且从目标方案看，这也不应该成为主设计方向。

2. 当前的自适应仍然是“有限场景库内的分支与推荐”，不是“大规模场景网中的实时叙事生成”。
   已有的是主线 + 少量支线 + 推荐策略；还不是多层级、多角色、多阶段的大型训练内容网络。

3. `identity` 目前更像玩家画像字段，不是领域主事实源。
   当前没有证据表明身份会驱动不同训练目标、不同场景库、不同评分标准或不同报告模型。

4. 后端收口还没完全结束。
   根据当前 review，`diagnostics` 的恢复语义还没和其他 read model 完全统一，training route 还残留通过 message 文本推断错误码的过渡逻辑，真实 DB 自检与 smoke 链路也还需要最后验证。

## 对“完整训练模式”的正确理解

如果按当前代码和目标文档综合判断，项目的“完整训练模式”更应该被定义为：

1. 玩家以战地记者训练身份进入固定历史时间轴。
2. 系统按 6 个核心历史场景组织训练主线。
3. 系统可在主线之间插入分支、补救、强化或推荐场景。
4. 玩家决策会影响风险状态、分支路径、补救链路、能力成长和最终结局。
5. 固定历史事件与历史结论不被改写。
6. 系统最终输出结构化训练报告与复盘结果。

而不应该被定义为：

1. 由 LLM 自由生成随机历史故事。
2. 玩家可任意身份进入完全不同的大型叙事宇宙。
3. 历史主线可被玩家随意改写。

## 如果要走向你描述的更强版本，还缺什么

### 前端

1. 明确“训练会话事实源”到底由谁权威持有，并完成恢复路径收口。
2. 把训练页中的视图投影、恢复决策、提交编排继续拆分，避免巨型 flow。
3. 把报告、进度、诊断三类 read-model 消费统一成稳定壳层。
4. 如果未来真要支持“其他身份”，前端必须先有角色差异化入口、流程和展示语义，而不是只传一个 `identity` 字段。

### 后端

1. 先把当前训练后端收口完成，包括 query/read-model 恢复一致性、错误契约类型化、真实 DB smoke 与自检闭环。
2. 扩充场景库和推荐策略，把“少量分支”演进为“可控的多层训练路径”。
3. 如果未来要引入更强 LLM 参与，也应该让 LLM 负责受约束的评估、生成局部反馈或辅助表达，而不是直接成为无边界故事事实源。
4. 如果未来真要支持多身份训练，必须补独立的角色目标、场景池、评分 rubrics 和报告模型，不能只靠 `player_profile.identity`。

## 最终结论

你这句话里，正确的部分是：

1. 有 6 个核心大场景。
2. 决策会影响后续路径。
3. 历史大事件是固定锚点。
4. 最后会生成训练报告。

不正确或容易误导的部分是：

1. 当前不是“LLM 推理出的随机故事线”。
2. 当前也不是“记者或其他身份都已经具备完整差异化主线”。

因此，当前项目更准确的定义应该是：

`抗战记者训练主线 + 配置化分支 + 风险/能力状态驱动 + 可选 LLM 评估 + 最终训练报告`
