# 训练主线业务说明

## 1. 业务概述

玩家从训练入口进入训练身份设定页，完成角色形象与基础身份信息填写后，系统创建训练会话，并进入首个训练场景。训练主线以故事化方式持续推进，玩家会在多个阶段化场景中，根据后端返回的候选选项，或结合场景上下文输入自由行动来完成每一轮决策。

每一轮决策都会提交到后端。后端基于当前会话的 `k_state / s_state`、风险标记、阶段策略、推荐结果和分支策略更新训练状态，并据此决定后续场景的推进方向、可选候选场景以及整体训练节奏。随着训练持续进行，系统逐步沉淀回合记录、状态变化、决策上下文、后果事件和阶段变化等训练数据。

当训练主线完成后，系统会生成本次训练的结局结果、训练报告和诊断视图，帮助玩家复盘自己在整条训练链路中的选择、能力变化、风险表现和整体完成情况。

## 2. 更贴合当前架构的表达

如果需要在产品文档、接口文档或流程说明中使用，建议采用下面这版标准表述：

> 玩家在训练入口进入训练身份设定页，完成角色形象与基础身份信息后，系统创建训练会话并进入第一个训练场景。训练主线以故事化方式连续推进，玩家会在多个阶段化场景中，根据后端返回的候选选项或自由输入进行决策。每一轮决策都会提交到后端，后端基于当前 `k_state / s_state`、风险标记、阶段策略、推荐结果和分支策略更新训练状态，并决定后续场景与训练推进方向。训练完成后，系统生成结局结果、训练报告和诊断视图，供用户复盘本次训练表现。

## 3. 术语边界

为避免和故事主线混淆，当前训练文档建议固定使用以下术语：

- 会话主键使用 `session_id`，前端对应 `sessionId`
- 训练是独立训练主线，不表述为“衔接 story 主线”
- 对外稳定场景字段优先使用 `scenario` / `scenario_candidates`
- 状态驱动字段优先使用 `k_state`、`s_state`、`decision_context`、`consequence_events`

## 4. 为什么这版更契合当前实现

这版表述与当前实现更一致，主要体现在：

- 前端确实通过训练入口页完成训练身份设定并启动训练会话
- 后端确实通过回合提交、状态更新和下一场景选择驱动训练推进
- 训练完成后确实提供报告与诊断读取路径
- 表述中不再把训练会话和故事会话混为一套概念

## 5. 关于“不是单一 KS 算法，而是多因素共同驱动”的详细思路

当前训练主线的推进机制，不适合再被简化描述成一个单一的“KS 算法”。更贴合当前实现的说法是：训练推进由一组相互配合的状态与策略共同驱动，其中 `k_state / s_state` 是底层状态事实源，风险标记、阶段策略、推荐结果和分支策略是在这些状态之上工作的决策层。

### 5.1 `k_state` 的作用

`k_state` 主要承载能力成长维度，例如知识、判断、核验、采访处理、风险应对等训练能力项。它的核心作用不是直接决定“剧情往哪走”，而是回答“玩家当前在哪些能力上偏弱、偏强、是否有成长”。

在当前实现里，`k_state` 主要用于：

- 记录每轮训练后的能力状态变化
- 作为推荐策略判断“应该优先补哪类能力”的依据
- 作为训练报告中能力雷达图、成长曲线和综合得分的核心输入
- 作为结局判定中的能力基础分

也就是说，`k_state` 更像训练能力面的长期状态，而不是一个直接控制剧情跳转的单点算法变量。

### 5.2 `s_state` 的作用

`s_state` 主要承载训练运行中的情境状态、风险状态和叙事状态，例如公众恐慌、信源安全、编辑信任、准确性等。它反映的不是“玩家会了多少”，而是“当前局面已经被玩家推进到什么状态”。

在当前实现里，`s_state` 主要用于：

- 反映本轮行动对训练局面的即时影响
- 驱动后果事件和运行时状态条的变化
- 参与推荐策略中的风险补救权重计算
- 参与最终结局判定，例如失败、稳健、恢复、优秀等类型判断

因此，`s_state` 更接近故事化训练中的运行时情境状态，是训练推进中的“局面事实源”。

### 5.3 风险标记的作用

风险标记来自每轮评估过程中的结构化输出，例如 `risk_flags`。它负责把“本轮做错了什么、触发了什么风险、是否已经进入高风险路径”显式沉淀下来，而不是只留在自然语言 message 里。

在当前实现里，风险标记主要用于：

- 对本轮状态更新施加额外惩罚或权重修正
- 进入诊断与审计视图，帮助后续复盘
- 影响后续推荐策略，推动系统优先给出补救型场景
- 参与结局判定，避免出现“总分还行但过程高风险”的误判

所以风险标记的价值，不是单轮提示，而是把训练过程里的危险路径结构化、可追踪地保存下来。

### 5.4 阶段策略的作用

阶段策略负责控制训练节奏，回答“当前训练应该处于开场、中段、高潮还是收束阶段”。这部分能力当前不是页面层临时拼装出来的，而是后端训练域中的独立策略能力。

阶段策略的核心作用有两层：

- 控制训练节奏，不让场景推进失去层次
- 让推荐和分支发生在合理阶段，而不是只按静态题库顺序乱跳

从业务上看，阶段策略让训练主线具备“故事化推进”的节奏感；从架构上看，它把“训练节奏”从散落的硬编码里抽离出来，收敛成可解释、可扩展的领域规则。

### 5.5 推荐结果的作用

推荐结果不是简单地“给用户一个下一题”，而是当前训练系统用于组织候选场景和决策上下文的关键中间层。

它主要承担三件事：

- 在 `adaptive` 模式下决定最优下一场景
- 在 `self-paced` 模式下给出推荐场景和候选场景池
- 把“系统为什么推荐这个场景、用户最后选了什么”沉淀成 `decision_context`

这意味着推荐结果本身既服务于训练推进，也服务于后续复盘和诊断。它不是一次性临时计算结果，而是训练过程中可被追溯的决策痕迹。

### 5.6 分支策略的作用

分支策略负责把“状态变化”转化为“场景路径变化”。如果说 `k_state / s_state` 描述的是当前状态，那么分支策略描述的就是“在这个状态下，训练接下来应进入哪条路径”。

它的业务作用主要体现在：

- 根据命中的规则把训练导向不同场景
- 在需要时生成 `branch_transition`
- 让训练主线不再只是线性题目序列，而是具备条件分支能力
- 把场景跳转原因、命中规则和触发标记保留下来，供报告与诊断使用

因此，分支策略是训练从“顺序做题”升级为“状态驱动的故事化训练推进”的关键一层。

### 5.7 五类驱动项之间的关系

把这五类因素合起来看，当前训练推进逻辑更接近下面这条链路：

1. 用户提交本轮决策。
2. 后端评估本轮输入，得到能力变化、状态变化和风险标记。
3. 系统更新 `k_state / s_state`，形成新的训练状态。
4. 系统结合阶段策略、推荐策略和分支策略，决定下一步训练推进方式。
5. 训练完成时，再基于累计状态、风险路径和结果数据生成结局与报告。

所以更准确的架构口径是：

> 当前训练系统不是由单一“KS 算法”驱动，而是由 `k_state / s_state` 作为状态事实源，再叠加风险标记、阶段策略、推荐结果和分支策略，共同决定训练推进与结果生成。

## 6. 关于“最终产出是结局结果、训练报告、诊断视图”的详细思路

当前训练主线的最终输出，不应该只被描述为“生成一个训练报告”。更准确的说法是：训练完成后，系统会形成三层产出，它们共用同一个 `session_id` 事实源，但承担不同的业务职责。

### 6.1 第一层产出：结局结果

结局结果是训练主线收束时的结果判定，它回答的是：

- 这次训练最后落到了什么结局类型
- 为什么会得到这个结果
- 训练整体能力和风险状态最终处于什么水平

结局结果更偏“训练收尾判定”，是训练是否完成、完成得如何的最终结论层。它的特点是：

- 面向训练闭环收束
- 强调最终判定与解释
- 通常比完整报告更轻、更聚焦结果

从产品视角看，结局结果解决的是“这次训练最后走到了哪里”的问题。

### 6.2 第二层产出：训练报告

训练报告是面向玩家复盘的主输出，它不只告诉玩家“最后结果”，还回答：

- 训练前后能力和状态发生了什么变化
- 哪些能力提升明显，哪些能力仍是短板
- 每一轮做了什么、状态如何变化
- 整体训练过程的成长轨迹是什么样的

在当前实现里，训练报告已经不是一个简单摘要，而是一个结构化读模型，主要包含：

- 总结摘要
- 能力雷达图
- 状态雷达图
- 成长曲线
- 历史回合记录
- 复盘建议

从业务上看，训练报告是“面向用户复盘”的主视图，是训练完成后最核心的结果阅读入口。

### 6.3 第三层产出：诊断视图

诊断视图不是给用户看结局的，它更偏向“解释系统过程”和“分析训练质量”。它重点回答：

- 系统推荐过什么、用户最终选了什么
- 训练过程中发生过哪些审计事件
- 哪些轮次出现了高风险或路径偏移
- 哪些能力或风险项被反复触发
- 阶段切换和分支跳转是如何发生的

在当前实现里，诊断视图的核心组成包括：

- `recommendation_logs`
- `audit_events`
- `kt_observations`
- 诊断摘要统计

从产品与架构角度看，诊断视图的价值在于，它把训练系统内部的“为什么会这样推进”解释出来，而不是只给用户一个最终成绩。

### 6.4 为什么要拆成三层产出

如果只输出一个“训练报告”，会把三类不同问题混在一起：

- 结果判定问题
- 用户复盘问题
- 系统诊断问题

当前拆成“结局结果 + 训练报告 + 诊断视图”三层，是为了让每类信息各自承担明确职责：

- 结局结果负责收束训练闭环
- 训练报告负责用户复盘与成长展示
- 诊断视图负责过程解释、风险分析和内部诊断

这也是为什么当前后端会分别提供不同读取模型，而不是把所有内容塞进一个超大响应里。

### 6.5 三层产出之间的关系

这三层产出虽然职责不同，但它们并不是三套彼此独立的数据源，而是围绕同一个训练会话沉淀出来的不同读模型。

它们之间的关系可以概括为：

- 结局结果：给出最终判定
- 训练报告：组织完整复盘结果
- 诊断视图：解释训练过程为什么会这样发生

因此，更适合写入文档的口径是：

> 当前训练系统的最终输出不是单一“训练报告”，而是围绕同一训练会话形成三层结果模型：结局结果负责训练收束，训练报告负责用户复盘，诊断视图负责过程解释与风险分析。

## 7. 建议写入正式文档的标准说法

如果需要把上面两点合并成一段正式描述，建议使用下面这版：

## 8. KS 算法设计的完整性评估

如果把“KS 算法”理解为当前训练系统里围绕 `k_state / s_state` 的整套状态评估、状态更新、路径选择、结果生成和复盘闭环，那么从当前实现来看，它已经不是一个零散原型，而是一套具备较强工程完整性的训练状态驱动体系。

但这里要先明确一个判断标准：

- 如果按“工程闭环是否完整”来评估，当前完整性较高
- 如果按“是否已经沉淀为一个统一、单体、可形式化证明的 KS 算法模型”来评估，当前还不算完全收口

因此，更合理的结论是：

- 工程闭环完整性：约 `85% - 90%`
- 统一算法模型完整性：约 `70% - 75%`

这两个分数不是矛盾的，而是分别回答两个不同问题：

- 第一类问题是“它能不能稳定跑完整条训练链路”
- 第二类问题是“它是不是已经抽象成一个单一、统一、标准化的 KS 理论模型”

### 8.1 为什么工程闭环完整性较高

从当前代码实现看，KS 相关能力已经覆盖了训练系统里最关键的 8 个环节。

#### 1. 状态定义完整

当前训练系统已经明确区分了两类状态：

- `k_state`：能力成长面
- `s_state`：运行时情境与风险面

并且这两类状态不是临时散落字段，而是有默认值、代码集合和统一来源的稳定结构。当前实现里已经有：

- 默认 `K/S` 状态
- 稳定的 `skill_codes`
- 稳定的 `s_state_codes`
- 配置驱动的默认值加载

这意味着系统已经完成了 KS 状态模型的“基础建模层”。

#### 2. 评估链路完整

当前每轮训练输入并不是只做一次简单打分，而是已经有完整评估链路：

- 规则评估
- 可选 LLM 评估
- 规则与 LLM 融合
- `skill_delta / s_delta` 结构化输出
- `risk_flags` 风险标记输出
- 置信度与 fallback 语义
- 高风险硬惩罚修正

这说明 KS 系统已经具备“从输入到结构化状态变化”的核心转换能力，而不是只停留在题库跳转层。

#### 3. 状态更新与快照完整

KS 系统当前不是只更新内存态，而是已经把状态变化沉淀为稳定快照与可回放数据：

- 初始 `K/S` 状态会落库
- 每轮提交后会写入新的状态结果
- `k_state` 快照会持久化
- `s_state` 快照会持久化
- 历史查询、报告和诊断都可以回读这些状态结果

这意味着 KS 系统已经具备“状态可追踪、可恢复、可复盘”的能力，这一点对训练系统是非常关键的完整性标志。

#### 4. 场景推进逻辑完整

当前训练推进并不是“用户提交后直接跳下一题”，而是已经形成了完整的路径选择层，包括：

- `guided / adaptive / self-paced` 三种模式
- 推荐排序
- 候选场景列表
- 提交校验
- 强制轮次控制
- 阶段推进
- 分支跳转

也就是说，KS 不是只影响“得分”，而是已经实质参与“训练接下来怎么走”。

#### 5. 结果收束完整

当前训练系统不是只更新过程状态，不知道如何结束；它已经具备完整的结果收束层：

- 基于 `k_state / s_state` 的结局判定
- 训练报告生成
- 诊断摘要生成
- 复盘建议生成

这说明 KS 系统已经从“过程计算”延伸到了“结果生产”。

#### 6. 可解释性完整

当前训练状态变化并不是黑箱结果，系统已经为可解释性保留了结构化中间产物：

- `decision_context`
- `recommendation_logs`
- `audit_events`
- `kt_observations`
- `consequence_events`
- `branch_transition`
- `phase_transition`

这意味着训练系统不仅知道“结果是什么”，还保留了“为什么会这样”的解释数据。

#### 7. 诊断与观测完整

当前 KS 驱动链路已经进入诊断和分析层，而不是只给前端一个结果页面：

- 能统计高风险轮次
- 能统计推荐偏差
- 能统计阶段切换
- 能统计分支跳转
- 能统计重点能力与风险项

这表明 KS 系统已经具备“可观测性”，这是工程完整性的一个核心标准。

#### 8. 配置化程度较高

当前很多与 KS 推进相关的规则并不是硬编码写死在主服务里，而是已经被抽离为：

- 运行时配置
- recommendation policy
- phase policy
- ending policy
- reporting policy

这说明系统已经从“写死规则”进化为“策略化、配置化、可扩展”的结构。

### 8.2 为什么统一算法模型完整性还没有完全收口

虽然工程闭环已经比较完整，但如果从“是不是一个统一 KS 算法模型”的角度看，当前还没有完全达到那种程度。

主要原因在于：

#### 1. 当前更像“多策略协同体系”，而不是“单一算法内核”

现在的训练推进是由多个模块共同完成的：

- evaluator
- recommendation policy
- phase policy
- branch resolver
- ending policy
- reporting policy

这在工程上是合理的，但也意味着“KS 算法”并不是一个单体函数或单个统一推理核，而是一个分层策略系统。

#### 2. `k_state` 和 `s_state` 的角色已经清晰，但统一理论抽象还不够集中

当前代码里，`k_state` 负责能力面，`s_state` 负责情境面，这个边界是清楚的；但它们之间的相互作用，仍然主要通过多处策略协作体现，而不是被抽象成一套统一的状态转移理论模型。

换句话说：

- 当前实现“能跑、能解释、能复盘”
- 但还没有完全抽象成“一个统一 KS 状态机理论”

#### 3. 参数校准更多是配置驱动，而不是学习驱动

目前的权重、boost、阈值、阶段窗口和结局阈值，主要来自配置和策略规则。这种设计对当前阶段是合理的，因为它更稳定、更可控，但这也说明 KS 体系目前更偏“规则化工程实现”，而不是“自适应学习型算法系统”。

#### 4. 报告与诊断已经很强，但它们更多是消费 KS 结果，而不是反向校准 KS 内核

当前报告和诊断已经可以很好地展示 KS 结果，但从现有实现来看，它们主要承担的是：

- 结果展示
- 过程解释
- 风险分析

而不是把这些结果再自动回流到算法调参或策略自更新中。所以从“算法闭环自学习”角度看，还没有完全封口。

### 8.3 当前最准确的完整性结论

因此，如果要把“KS 算法设计完整性”写进正式文档，建议采用下面这版口径：

> 当前训练系统中的 KS 设计，已经具备较高的工程闭环完整性。它不仅完成了 `k_state / s_state` 的状态建模，还打通了每轮输入评估、状态更新、风险沉淀、推荐排序、阶段推进、分支跳转、结局判定、训练报告和诊断视图，形成了一条可运行、可恢复、可解释、可复盘的训练主链路。从工程落地视角看，完整性约为 `85% - 90%`。但如果从“是否已经收敛为统一单体 KS 算法模型”的角度评估，当前实现仍更接近“多策略协同的状态驱动体系”，统一算法模型完整性约为 `70% - 75%`。

### 8.4 这意味着什么

这个结论的真正含义不是“当前设计不完整”，而是：

- 作为训练系统，它已经足够完整，能支撑真实训练流程
- 作为算法产品，它已经不是简单原型，具备较强可扩展性
- 但如果未来要把 KS 单独包装成“方法论”或“核心算法品牌”，还需要继续统一抽象层

更直白地说：

- 现在的 KS 设计，已经足够支撑训练系统上线和持续扩展
- 但还没有收敛成一个可以单独命名、单独输出白皮书的统一算法内核

## 9. 如果要把 KS 做成可命名、可白皮书化的统一算法内核，后续优化方案

如果目标不只是“让训练系统能运行”，而是要把 KS 单独沉淀成可以命名、可以对外讲、可以形成方法论文档甚至白皮书的核心算法能力，那么后续优化的重点就不再只是继续加功能，而是要把当前分散在多个策略里的能力收口成一套统一抽象。

当前最大的问题不是能力不够，而是能力分布在多个模块中：

- evaluator 负责评估
- recommendation policy 负责推荐
- phase policy 负责阶段
- branch resolver 负责分支
- ending policy 负责结局
- reporting policy 负责报告与诊断聚合

这套结构工程上是合理的，但如果想对外说“我们有一个 KS 核心算法”，那就需要进一步回答下面这几个问题：

1. KS 的输入到底是什么
2. KS 的状态到底是什么
3. KS 的状态转移规则到底是什么
4. KS 的输出到底有哪些层级
5. KS 的效果如何验证
6. KS 与业务策略的边界在哪里

只有这些问题都被统一定义，KS 才能从“多策略协同体系”变成“统一算法内核”。

### 9.1 目标状态

如果要做到“可单独命名、可单独输出白皮书的统一算法内核”，建议把目标状态定义成下面 5 条：

#### 1. 有统一算法名字

KS 不再只是项目内部对 `k_state / s_state` 的简称，而是形成一个明确可复用的算法名称，例如：

- `KS State Engine`
- `KS Narrative Training Engine`
- `KS Decision Progression Model`

重点不是先取什么名字，而是先让它具备“名字背后有稳定定义”的能力。

#### 2. 有统一输入输出模型

对外能清楚说明：

- 输入：用户决策、场景配置、历史状态、风险上下文
- 中间状态：`k_state / s_state / risk_flags / phase_snapshot / branch_context`
- 输出：状态更新、推荐结果、路径跳转、结局结果、复盘结果

也就是说，KS 必须有一张明确的“状态转移图”和“输入输出契约图”，不能再靠代码阅读者自己拼。

#### 3. 有统一状态转移逻辑

当前状态变化是分布在多个模块里的，未来应进一步统一成一条可表达的核心逻辑链：

`输入 -> 评估 -> 状态增量 -> 风险修正 -> 阶段解析 -> 推荐排序 -> 分支决策 -> 结果收束`

这条链路一旦统一，KS 才能真正从“实现集合”收敛成“方法论”。

#### 4. 有统一验证基线

KS 要成为算法产品，不能只说“逻辑合理”，还要有清晰验证方法，例如：

- 固定输入的状态更新是否稳定
- 推荐排序是否满足预期
- 高风险路径是否会被纠正
- 不同模式下是否产生一致可解释结果
- 报告和诊断是否能正确解释状态变化

没有验证基线，就很难把 KS 做成可对外讲的核心算法资产。

#### 5. 有统一文档体系

至少要形成 4 类文档：

- 术语定义文档
- 算法框架文档
- 验证与评估文档
- 业务落地案例文档

这样才能从“代码里有一套东西”升级成“对内可复用、对外可介绍的算法产品”。

### 9.2 当前差距

当前距离“统一算法内核”还有 6 个主要差距。

#### 1. 缺少单一 KS 核心抽象层

现在有 `k_state / s_state`，但没有一个明确的 `KSKernel` 或 `KSCore` 层，把状态输入、状态转移、策略输出统一组织起来。

#### 2. 状态转移规则分散在多个 policy 中

这意味着当前系统“能工作”，但很难一句话说清楚：

- KS 到底如何更新
- 哪些规则是核心算法
- 哪些规则只是业务策略

#### 3. 核心算法与业务模式边界仍不够清晰

例如：

- `guided`
- `adaptive`
- `self-paced`

这些模式当前和推荐逻辑、提交校验、候选场景构建耦合得比较紧。未来如果要形成“KS 方法论”，需要先区分：

- 哪些是 KS 内核能力
- 哪些是产品模式策略

#### 4. 缺少统一算法指标体系

当前已经有报告和诊断，但还没有一套显式的“算法评估指标”文档，例如：

- 推荐命中率
- 风险纠偏率
- 状态变化稳定性
- 结局区分度
- 复盘解释一致性

#### 5. 缺少 benchmark 和回归基线

当前测试更偏接口与功能回归，还没有形成专门面向 KS 算法的 benchmark 数据集与标准回放场景。

#### 6. 缺少对外方法论包装层

当前能解释“代码怎么做”，但还不能很好解释：

- KS 的核心理念是什么
- 为什么比传统线性题库训练更强
- 为什么它能支持故事化训练
- 它的适用边界是什么

### 9.3 建议的后续优化路径

建议按 4 个阶段推进，而不是一次性重构。

### 第一阶段：统一术语与核心边界

目标：先把 KS 说清楚。

建议动作：

1. 固定 KS 的正式定义。
   建议明确：
   - `K` 表示能力成长面
   - `S` 表示情境与风险面
   - KS 不是单一打分器，而是状态驱动训练引擎

2. 明确 KS 内核与业务策略的边界。
   建议拆成两层：
   - KS 内核：状态更新、风险修正、阶段快照、路径计算
   - 业务策略：模式配置、场景内容、产品展示、报告样式

3. 输出统一术语表。
   至少固定下面这些概念：
   - `state`
   - `state_delta`
   - `risk_signal`
   - `phase_snapshot`
   - `recommendation_result`
   - `branch_transition`
   - `terminal_outcome`

这一阶段完成后，最大的收益是：团队内部对 KS 的理解会统一，后续重构不会边改边重新定义概念。

### 第二阶段：建立统一 KS Core 抽象层

目标：把当前分散在多处的能力收口到一个核心算法层。

建议做法：

1. 在训练域中新增一层显式核心模块，例如：

```text
backend/training/ks_core/
  state_model.py
  input_model.py
  transition_model.py
  kernel.py
  output_model.py
  metrics.py
```

2. 统一定义一个核心执行接口，例如：

```python
result = ks_kernel.run(
    session_state=...,
    round_input=...,
    scenario_context=...,
    mode_context=...
)
```

3. 让现有模块逐步变成 KS Core 的子能力，而不是平铺在 service/policy 外围。
   例如：
   - evaluator 输出标准化 `state_delta`
   - phase policy 输出标准化 `phase_snapshot`
   - recommendation policy 输出标准化 `recommendation_result`
   - branch resolver 输出标准化 `branch_transition`
   - ending policy 输出标准化 `terminal_outcome`

4. 最终由 `kernel.py` 统一组装成一条完整转移链。

这一阶段完成后，KS 才真正具备“有一个核心引擎”的形态。

### 第三阶段：建立算法验证与 benchmark 体系

目标：让 KS 不只是可运行，还可证明、可比较、可调优。

建议动作：

1. 建立标准训练回放样本集。
   每条样本至少包括：
   - 初始 `k_state / s_state`
   - 用户输入序列
   - 预期风险变化
   - 预期推荐结果
   - 预期结局范围

2. 建立核心评估指标。
   建议至少覆盖：
   - 状态更新稳定性
   - 推荐排序合理性
   - 风险识别准确性
   - 分支跳转一致性
   - 结局判定区分度
   - 报告解释一致性

3. 增加 KS 专项测试层。
   当前已有功能测试，但后续应补：
   - `test_ks_kernel.py`
   - `test_ks_benchmark.py`
   - `test_ks_state_transition.py`
   - `test_ks_recommendation_consistency.py`

4. 给每次策略调参建立回归报告。
   每次权重、阈值、phase boost 或 risk boost 变更，都应有前后对比结果，而不是只看是否“感觉更合理”。

这一阶段完成后，KS 才具备算法产品应有的评估基础。

### 第四阶段：方法论与品牌化沉淀

目标：把 KS 从工程实现沉淀为对外可讲的方法论。

建议产出 5 类内容：

#### 1. 一份 KS 核心算法说明文档

它回答：

- KS 解决什么问题
- KS 的输入、状态、转移、输出是什么
- KS 与传统题库式训练的区别是什么

#### 2. 一份 KS 评估与验证文档

它回答：

- KS 如何被验证
- KS 的核心指标是什么
- KS 的优势如何量化

#### 3. 一份 KS 业务落地案例文档

它回答：

- 在训练系统里，KS 如何驱动场景推进
- 如何影响推荐、风险和结局
- 玩家最终能看到什么价值

#### 4. 一份 KS 白皮书大纲

建议至少包含：

1. 问题定义
2. 训练系统挑战
3. KS 方法论
4. 状态模型设计
5. 状态转移与路径推进
6. 风险与阶段控制
7. 结果生成与可解释性
8. 实验与案例

#### 5. 一套可复用的品牌表达

例如统一三句话：

- KS 是一套状态驱动的故事化训练推进引擎
- KS 用能力状态和情境状态共同驱动训练路径
- KS 不只给结果，还保留训练过程的解释能力

这一阶段完成后，KS 才能真正从“项目内部实现”升级成“可对外传播的核心算法产品”。

### 9.4 建议优先级

如果资源有限，建议按下面优先级做，而不是一上来就写白皮书。

优先级 1：

- 统一术语
- 拆出 KS Core 抽象层
- 明确内核与业务策略边界

优先级 2：

- 建 benchmark
- 建专项测试
- 固化核心指标

优先级 3：

- 输出方法论文档
- 输出白皮书与品牌表达

原因很简单：

- 没有统一抽象层，白皮书容易变成对现有代码的事后解释
- 没有 benchmark，方法论难以自证
- 没有指标体系，品牌表达就容易停留在口号层

### 9.5 可以直接写进规划文档的结论

如果要把这部分写成正式规划结论，建议采用下面这版：

> 如果后续目标是把 KS 从训练系统内部能力进一步沉淀为可命名、可方法论化、可白皮书化的核心算法产品，后续优化重点不应只是继续扩展功能，而应优先完成统一术语、统一状态转移抽象、统一验证指标和 benchmark 基线建设。在此基础上，再将 evaluator、推荐、阶段、分支、结局与报告能力逐步收敛到统一 KS Core 抽象层，最终形成“输入明确、状态明确、转移明确、输出明确、验证明确”的统一算法内核。只有完成这一步，KS 才能真正从“可运行的训练状态驱动体系”升级为“可独立命名、可独立对外输出的方法论与核心算法品牌”。
