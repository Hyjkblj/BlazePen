# 烽火笔锋训练引擎升级为游戏运行时后端开发计划

## 1. 文档目标

本计划用于把当前已经落地的“训练流水线后端”升级为“可运行的叙事决策游戏后端”。

当前后端已经具备：

1. 训练会话初始化
2. 场景推进与回合提交
3. K / S 状态更新
4. 评估、结局、报告、诊断闭环
5. 持久化与回放能力

但当前仍偏向：

- `Evaluator Pipeline`
- `Training Workflow`

而不是完整的：

- `Runtime Engine`
- `Consequence-driven Narrative System`

所以本轮后端开发的核心目标不是推翻现有训练架构，而是：

**在现有 TrainingService / Policy / Repository / DTO 体系之上，增量补出“运行时状态 + 后果驱动 + 分支叙事”这一层。**

---

## 2. 核心判断

### 2.1 当前系统的本质

当前系统最擅长的是：

1. 接收玩家输入
2. 做规则 / LLM 评估
3. 更新能力状态和叙事状态
4. 生成报告与结局

当前系统最缺的是：

1. 显式的游戏运行时状态对象
2. 显式的后果驱动系统
3. 显式的分支场景图
4. 显式的“世界变化”表达

一句话总结：

**系统现在会判断，但还不会让世界发生变化。**

### 2.2 本轮升级原则

本轮升级必须遵守以下原则：

1. 不重写 `TrainingService`
2. 不新造一套平行状态系统
3. 不把当前训练主链路打散
4. 优先复用已有的 `K / S / session_meta / snapshot / DTO / Policy`
5. 先做最小可运行 Runtime，再考虑大规模 Scene Graph 和 AI 导演化

---

## 3. 升级后的目标架构

当前架构：

```text
Player Input
    ↓
TrainingService
    ↓
Evaluator
    ↓
K / S State Update
    ↓
Store / Report / Ending
```

目标架构：

```text
Player Input
    ↓
TrainingService
    ↓
GameRuntimeAdapter
    ↓
ConsequenceEngine
    ↓
Runtime State Update
    ↓
Scene Flow / Branch Resolution
    ↓
Store / Report / Ending / Runtime Events
```

新增的关键层有 3 个：

1. `GameRuntimeState`
2. `ConsequenceEngine`
3. `Scene Graph / Branch Resolver`

---

## 4. 与现有后端的结合策略

## 4.1 可以直接复用的现有能力

以下内容不需要推翻，应该直接复用：

1. `TrainingSession.k_state`
2. `TrainingSession.s_state`
3. `TrainingSession.session_meta`
4. `NarrativeStateSnapshot`
5. `RoundEvaluation`
6. `TrainingRoundFlowPolicy`
7. `ScenarioRepository`
8. `EndingPolicy`
9. `TrainingTelemetryPolicy`
10. `TrainingReportOutput / DiagnosticsOutput`

## 4.2 本轮不建议直接大改的地方

以下内容本轮不应作为第一优先级：

1. 不先重构整个旧无限剧情流
2. 不先接 Redis / MQ
3. 不先引入 Chroma 到训练主链路
4. 不先把全部场景改成复杂图结构
5. 不先让 AI 接管完整剧情推进

原因很明确：

1. 当前瓶颈不在并发
2. 当前瓶颈不在检索
3. 当前最缺的是玩法闭环，不是基础设施花活

## 5. 总体优先级

## P0

目标：

**补出最小可运行的“游戏运行时内核”。**

内容：

1. 新增 `GameRuntimeState`
2. 新增 `ConsequenceEngine`
3. 把当前 `submit_round` 结果接入运行时后果解释
4. 把世界状态 flags 写入会话与快照
5. 让前端或 CLI 能拿到“世界变化结果”

## P1

目标：

**把“训练回合推进”升级为“运行时分支推进”。**

内容：

1. 给场景配置增加分支能力
2. 给回合流转策略增加分支解析器
3. 支持失败后果、锁定分支、补救分支
4. 让报告能回放“为什么进了这个分支”

## P2

目标：

**把 Runtime 变成完整的产品后端契约。**

内容：

1. 输出前端状态栏需要的稳定字段
2. 输出后果提示和世界事件
3. 支持更丰富的结局判定
4. 支持教师 / 运营视角的运行时复盘

## P3

目标：

**把 AI 从评分器升级为剧情辅助模块。**

内容：

1. AI 生成编辑部反馈
2. AI 解释玩家稿件对世界的影响
3. AI 动态 NPC 反馈
4. AI 自由输入稿件评分服务化

---

## 6. 分阶段开发计划

## 6.1 第一阶段：Runtime 基础层

优先级：

- `P0`

目标：

- 不改变当前训练闭环的前提下，引入显式运行时状态

新增模块建议：

1. `backend/training/runtime_state.py`
2. `backend/training/consequence_engine.py`
3. `backend/training/runtime_events.py`

### 6.1.1 GameRuntimeState

职责：

1. 聚合当前会话运行时状态
2. 统一表达“当前场景 + K 状态 + S 状态 + 世界 flags + 玩家档案”
3. 成为后续后果引擎和分支引擎的共同输入

建议结构：

```python
GameRuntimeState(
    session_id,
    current_round_no,
    current_scene_id,
    k_state,
    s_state,
    player_profile,
    runtime_flags,
)
```

其中 `runtime_flags` 第一版建议至少包括：

1. `panic_triggered`
2. `source_exposed`
3. `editor_locked`
4. `high_risk_path`

存储策略：

1. 第一版先放入 `training_sessions.session_meta.runtime_flags`
2. 同时在每轮审计 / 观测中保留快照
3. 暂不急着单独建一张 `runtime_flags` 表

### 6.1.2 ConsequenceEngine

职责：

1. 把评估结果解释成“世界后果”
2. 不只更新分数，还更新世界 flags
3. 生成结构化后果事件，供前端和报告消费

输入：

1. `GameRuntimeState`
2. 当前场景
3. 玩家选择
4. `RoundEvaluationPayload`

输出：

```python
ConsequenceResult(
    updated_runtime_state,
    triggered_flags,
    consequence_events,
    branch_hints,
)
```

第一版规则建议：

1. `source_exposure_risk` -> `source_exposed = True`
2. `high_risk_unverified_publish` 或 `public_panic` 超阈值 -> `panic_triggered = True`
3. `editor_trust` 低于阈值 -> `editor_locked = True`
4. 连续高风险轮次 -> `high_risk_path = True`

### 6.1.3 Runtime Event

职责：

1. 用结构化方式描述“这一轮世界发生了什么”
2. 避免前端和报告层重新猜测后果

第一版事件类型建议：

1. `editor_trust_dropped`
2. `public_panic_triggered`
3. `source_exposed`
4. `branch_locked`
5. `branch_recovered`

### 6.1.4 第一阶段需要修改的现有文件

1. `backend/api/services/training_service.py`
2. `backend/training/training_outputs.py`
3. `backend/training/telemetry_policy.py`
4. `backend/training/reporting_policy.py`
5. `backend/training/contracts.py`

### 6.1.5 第一阶段验收标准

满足以下条件即视为完成：

1. `submit_round` 返回中新增稳定的 runtime 字段
2. 系统能明确返回本轮世界后果
3. `session_meta` 中可看到 `runtime_flags`
4. `get_report / get_diagnostics` 可以回放世界后果
5. 原有训练测试不被打坏

---

## 6.2 第二阶段：Branch-ready Scene Flow

优先级：

- `P1`

目标：

- 把顺序题目流升级为“兼容分支的场景推进器”

新增或升级模块建议：

1. `backend/training/scene_graph.py`
2. `backend/training/branch_resolver.py`

### 6.2.1 场景配置升级

当前 `scenario_bank.json` 主要包含：

1. 场景正文
2. 任务目标
3. 风险标签
4. 预设选项

下一步建议增加：

1. `entry_conditions`
2. `next_rules`
3. `failure_routes`
4. `recovery_routes`
5. `branch_tags`

示例：

```json
{
  "id": "S3",
  "next_rules": [
    {
      "when_flag": "source_exposed",
      "go_to": "S3_FAIL_A"
    },
    {
      "when_flag": "panic_triggered",
      "go_to": "S4_RISK"
    },
    {
      "default": true,
      "go_to": "S4"
    }
  ]
}
```

### 6.2.2 RoundFlowPolicy 升级原则

当前 `TrainingRoundFlowPolicy` 的职责偏向：

1. 顺序推进
2. 推荐排序
3. 校验提交合法性

升级后增加：

1. 优先解析分支规则
2. 无分支时回退到现有顺序逻辑
3. 推荐模式仍能工作

原则：

**图结构优先，顺序流兜底。**

### 6.2.3 第二阶段验收标准

1. 至少支持 1 条失败支线
2. 至少支持 1 条补救支线
3. `get_next_scenario` 能基于运行时 flags 返回不同场景
4. 当前 6 幕主线还能正常跑通

## 6.3 第三阶段：Runtime API 契约稳定化

优先级：

- `P2`

目标：

- 让前端可以把当前系统真正当作“剧情游戏后端”消费

需要输出的新字段建议：

1. `runtime_state`
2. `runtime_flags`
3. `consequence_events`
4. `branch_hint`
5. `available_routes`
6. `state_bar`

其中 `state_bar` 建议直接输出前端可消费结构：

```json
{
  "editor_trust": 0.52,
  "public_stability": 0.68,
  "source_safety": 0.71
}
```

说明：

1. `public_stability` 可以由当前 `public_panic` 做镜像换算
2. 后端负责统一映射，前端不要自己硬算

### 6.3.1 API 层需要升级的接口

1. `init_training`
2. `get_next_scenario`
3. `submit_round`
4. `get_progress`
5. `get_report`
6. `get_diagnostics`

### 6.3.2 第三阶段验收标准

1. 前端无需自己拼世界状态
2. 状态栏字段有稳定 schema
3. 后果提示字段有稳定 schema
4. 报告可解释“这条剧情为什么发生”

---

## 6.4 第四阶段：AI 服务升级

优先级：

- `P3`

目标：

- 把 AI 从“评卷老师”升级为“剧情辅助导演”

第一批建议能力：

1. AI 编辑部反馈文案
2. AI 风险解释文案
3. AI 玩家稿件结构化点评
4. AI NPC 动态反馈

第二批建议能力：

1. 玩家自由输入报道稿
2. AI 给出风险、准确性、煽动性、来源保护评分
3. AI 输出结构化剧情调节建议

边界原则：

1. AI 不直接决定核心状态
2. AI 只提供解释、建议和扩展反馈
3. 核心世界变化仍由规则化 ConsequenceEngine 决定

这样做的原因：

1. 保证可测
2. 保证可控
3. 保证政治正确和稳定性

## 9. 数据存储策略建议

## 9.1 第一阶段

运行时状态存储采用轻量方案：

1. 主状态仍存 `training_sessions.k_state`
2. 主状态仍存 `training_sessions.s_state`
3. 世界 flags 先存 `training_sessions.session_meta.runtime_flags`
4. 每轮运行时后果通过 `audit_events` 和 `kt_observations` 留痕

这样做的好处：

1. 不破坏现有 schema 主结构
2. 回滚成本低
3. 不增加过多迁移复杂度

## 9.2 第二阶段

如果分支和事件变复杂，再评估是否新增：

1. `runtime_event_logs`
2. `branch_transition_logs`
3. `runtime_state_snapshots`

原则：

**先复用现有表，后期再拆专表。**

## 11. 风险与规避

## 风险 1

问题：

- 新增 `GameRuntimeState` 后形成双状态源

规避：

1. 运行时状态必须基于现有 `k_state / s_state / session_meta` 构建
2. 不允许再单独维护平行内存状态作为真实来源

## 风险 2

问题：

- 过早引入复杂 Scene Graph，打坏当前顺序训练流

规避：

1. 第一版只做“分支兼容层”
2. 无条件回退到现有顺序策略

## 风险 3

问题：

- AI 过早接管剧情推进，导致不可控

规避：

1. 核心后果先规则化
2. AI 只做反馈增强和文案层辅助

## 风险 4

问题：

- 世界后果只做了字段，没有形成前后端闭环

规避：

1. 后端必须输出结构化 `consequence_events`
2. 前端后续必须消费这些字段，而不是继续只展示报告

## 13. 一句话总结

这轮后端开发的真正目标不是继续堆训练接口，而是：

**把现有训练流水线升级成“会评估、会触发后果、会改变世界、会引导分支”的运行时叙事引擎。**

优先级上必须坚持：

1. 先 `GameRuntimeState`
2. 再 `ConsequenceEngine`
3. 再 `Branch Resolver / Scene Graph`
4. 最后再做 AI 导演化

只有按这个顺序推进，当前项目才能从“训练后端”真正走向“剧情决策游戏后端”。
