# 《烽火笔锋》二次开发数据库与业务架构设计（可落地版）

版本：v1.0  
日期：2026-03-13  
适用范围：`BlazePen` 现有代码仓库二次开发（训练系统化改造）

---

## 1. 目标与边界

### 1.1 业务目标

把当前“剧情互动系统”升级为“抗战历史情境下的战地记者决策与沟通能力训练系统”，实现：

1. 可训练：支持多轮决策训练与场景推进。
2. 可评估：每轮有结构化评分、证据链、能力变化。
3. 可成长：基于 KT（先 BKT，后 DKT）实现个性化推荐与成长报告。
4. 可解释：结局、评分、推荐都可回溯到证据与规则。

### 1.2 非目标

1. 不改写历史锚点与主结论。
2. 不作为真实新闻发布系统。
3. 不做医疗或心理诊断结论。

### 1.3 设计原则

1. 历史约束优先于剧情自由度。
2. 训练指标优先于“通关爽感”。
3. 规则与模型“双轨”评估，避免纯 LLM 漂移。
4. 关键决策可回放、可审计、可复盘。

---

## 2. 整体架构（业务 + 技术）

### 2.1 分层架构

1. 交互层（Frontend）  
训练页、回合反馈、报告页、雷达图、反事实对比页。

2. 应用层（API/Service）  
会话管理、场景推进、评分编排、KT 更新、推荐与结局判定。

3. 训练引擎层（Core Engine）  
历史约束引擎、Rubric 评分引擎、BKT/DKT 引擎、策略推荐引擎。

4. 数据层（Persistence）  
PostgreSQL（结构化训练数据）+ ChromaDB（语义记忆、证据检索）。

5. 治理层（Governance）  
审计日志、内容安全、模型版本追踪、离线回放。

### 2.2 核心数据流

1. 用户提交决策。
2. 历史约束校验（是否越界、是否违反锚点）。
3. 评分引擎生成 `skill_scores + evidence`。
4. 规则引擎做二次校准与惩罚/奖励。
5. KT 引擎更新 K1-K8 掌握度。
6. 剧情状态 S 更新并选择下一场景。
7. 生成本轮反馈与证据链。
8. 训练结束后生成报告与结局。

## 3. 数据库设计（成熟可扩展）

## 3.1 PostgreSQL 逻辑模型

### A. 历史与场景基座

1. `history_timeline_events`  
历史锚点事件表（不可改核心事实）。

2. `scenario_bank`  
训练场景主表（S1-S6 等）。

3. `scenario_decision_nodes`  
场景内决策节点（每轮核心决策点）。

4. `scenario_options`  
节点选项与预设影响（S 状态影响、风险等级、目标技能）。

### B. 训练过程主线

1. `training_sessions`  
一次完整训练会话（用户、阶段、状态）。

2. `training_rounds`  
每轮提交记录（输入、前后状态、反馈）。

3. `round_evaluations`  
每轮评分详情（LLM 输出、规则校准、证据链）。

### C. 能力追踪（KT）

1. `kt_observations`  
每轮每技能观测值（score、confidence、来源）。

2. `kt_state_snapshots`  
每轮能力快照（K1-K8 概率掌握度）。

### D. 剧情状态与推荐

1. `narrative_state_snapshots`  
每轮剧情状态 S 快照（可信度、准确性、恐慌等）。

2. `scenario_recommendation_logs`  
下一场景推荐日志（候选、utility 分解、最终选择）。

### E. 收口与审计

1. `ending_results`  
会话结局、解释文本、证据引用。

2. `training_audit_events`  
关键流程审计日志（可回放、可排错）。

### F. 向量索引映射

1. `memory_index_refs`  
结构化记录到 Chroma 文档 ID 的映射（便于追溯）。

## 3.2 与当前库的兼容关系

1. 复用现有 `characters` / `character_states` 作为人物与基础状态来源。
2. 新增训练表不破坏旧剧情系统，可并行运行。
3. 旧 `vector_db` 继续用于语义检索，但通过 `memory_index_refs` 建立可追踪映射。

## 3.3 关键索引策略

1. `training_rounds(session_id, round_no)`：高频查询回合轨迹。
2. `kt_state_snapshots(session_id, round_no DESC)`：快速取最近能力快照。
3. `kt_observations(session_id, skill_code, created_at)`：技能时序分析。
4. `scenario_bank(is_active, risk_level)`：候选场景筛选。
5. `round_evaluations` JSONB GIN：按证据关键词与规则标签检索。

## 4. ChromaDB 设计（语义层）

### 4.1 Collection 规划

1. `training_round_memory`  
单轮决策语义片段（用户输入 + 评分证据 + 情境摘要）。

2. `scenario_case_memory`  
场景案例库（典型正确/错误模式，供复盘和推荐）。

3. `history_reference_memory`  
历史约束与来源摘要（用于回答“为何不允许该分支”）。

### 4.2 文档结构建议

每条文档包含：

1. `document`：可向量化文本（情境摘要、决策文本、证据摘要）。
2. `metadata`：`session_id`、`round_id`、`scenario_id`、`skills`、`risk_level`、`model_version`。

## 5. 训练流程搭建（端到端）

## 5.1 会话初始化（`POST /api/v1/training/init`）

1. 创建 `training_sessions`。
2. 初始化 `kt_state_snapshots(round_no=0)`。
3. 初始化 `narrative_state_snapshots(round_no=0)`。
4. 基于弱项和课程目标选第一个场景。

## 5.2 获取下一场景（`GET /api/v1/training/scenario/next`）

1. 按时间轴和历史锚点过滤可选场景。
2. 按 `utility` 对候选场景排序。
3. 写入 `scenario_recommendation_logs`。
4. 返回当前节点与可选决策。

## 5.3 提交回合（`POST /api/v1/training/round/submit`）

1. 写 `training_rounds`（before 状态、输入、节点信息）。
2. 评分编排：LLM 结构化评分 + Rubric 规则校准。
3. 写 `round_evaluations` 与 `kt_observations`。
4. BKT 更新并写 `kt_state_snapshots`。
5. 更新剧情状态 S 并写 `narrative_state_snapshots`。
6. 若满足结束条件，进入结局生成。

## 5.4 训练报告（`GET /api/v1/training/report/{session_id}`）

1. 聚合 K1-K8 全过程曲线。
2. 汇总关键错误模式与证据链。
3. 输出反事实对比（若选项 A/B 的后果差异）。
4. 写入 `ending_results` 与报告缓存。

## 6. 算法实现（业务核心）

## 6.1 评分算法（LLM + 规则融合）

### 输入

1. 用户决策文本/选项。
2. 当前节点与历史上下文。
3. 历史约束条目与伦理规则。

### 输出（结构化）

```json
{
  "skill_scores": {
    "K1": {"score": 0.72, "confidence": 0.81, "evidence": ["..."]},
    "K5": {"score": 0.34, "confidence": 0.76, "evidence": ["..."]}
  },
  "risk_flags": ["source_exposure_risk"]
}
```

### 融合逻辑

`final_k = clip( w_llm * llm_k + w_rule * rule_k - w_penalty * penalty_k, 0, 1 )`

其中：

1. `llm_k`：模型评分。
2. `rule_k`：规则引擎评分（可解释强）。
3. `penalty_k`：硬性违规惩罚（如泄露线人信息）。

## 6.2 BKT 更新算法（冷启动主力）

对每个技能 `k` 维护掌握度 `P(L_k)`：

1. 正确观测时：
`P(L_k|obs=1) = P(L_k)(1-slip) / (P(L_k)(1-slip) + (1-P(L_k))guess)`

2. 错误观测时：
`P(L_k|obs=0) = P(L_k)slip / (P(L_k)slip + (1-P(L_k))(1-guess))`

3. 迁移更新：
`P(L_k,next) = P(L_k|obs) + (1-P(L_k|obs)) * T`

置信度融合建议：
`obs_weight = clamp(confidence, 0.3, 1.0)`，低置信度弱化更新幅度。

## 6.3 剧情状态更新算法（S 状态）

定义剧情状态向量 `S`：

1. `credibility`（公信力）
2. `accuracy`（准确性）
3. `public_panic`（公众恐慌）
4. `source_safety`（线人安全）
5. `editor_trust`（编辑部信任）
6. `actionability`（行动指引有效性）

更新函数：

`S_{t+1} = clamp( A * S_t + B * action_effect + C * risk_flags, 0, 1 )`

### 解释

1. `action_effect` 来自选项预设和规则引擎校正。
2. `risk_flags` 会触发高风险项的附加惩罚。

## 6.4 场景推荐算法（自适应）

`utility(scene_i) = α * info_gain + β * risk_priority + γ * weakness_cover - δ * cognitive_load`

因素说明：

1. `info_gain`：该场景能降低多少能力不确定性。
2. `risk_priority`：高风险能力（K1/K5/K6）优先权重。
3. `weakness_cover`：对当前短板技能覆盖程度。
4. `cognitive_load`：连续高压场景惩罚，避免训练失效。

## 6.5 结局判定算法

`ending_score = λ1 * mean(K1..K8) + λ2 * safety_index + λ3 * correction_ability - λ4 * severe_violation_count`

结局按阈值分段：

1. `史笔如铁`
2. `艰难守真`
3. `逆风修正`
4. `失真扩散`
5. `代价沉重`

## 7. 所有影响剧情的因素（全量清单）

| 因素类别 | 因素项 | 数据来源 | 对剧情影响 |
|---|---|---|---|
| 历史硬约束 | 时间范围、锚点事件、不可改结论 | `history_timeline_events` | 决定可用场景与可选分支上限 |
| 决策输入 | 玩家文本、选项、提交时机 | `training_rounds` | 触发节点推进、状态更新与评分 |
| 能力状态K | K1-K8掌握度与不确定性 | `kt_state_snapshots` | 影响推荐场景、反馈深度、结局评分 |
| 剧情状态S | 可信度/准确性/恐慌/线人安全等 | `narrative_state_snapshots` | 影响分支可见性、风险后果强度 |
| 评分证据 | LLM证据、规则标签、风险标志 | `round_evaluations` | 影响 KT 更新幅度与惩罚项 |
| 场景难度 | risk_level、target_skills、max_rounds | `scenario_bank` | 影响训练节奏和失败代价 |
| 用户历史 | 历史错误模式、复盘记录 | Chroma + SQL | 影响个性化推荐与提示方式 |
| 合规与安全 | 内容安全拦截、伦理规则命中 | 审计与规则引擎 | 触发强制改写、扣分或回退模板 |
| 系统置信度 | 模型 confidence、校准差异 | `round_evaluations` | 影响分数可信权重与二次审核 |
| 训练疲劳 | 连续高压轮次、认知负荷估计 | 推荐日志 + 会话统计 | 影响下一场景强度与节奏 |

## 9. 实施计划（建议）

### P1（1-2周）：可运行训练主线

1. 建表与迁移完成。
2. 单场景（S1）打通 `init -> submit -> progress -> report`。
3. BKT 基线版可用。

### P2（2-3周）：评分和推荐增强

1. LLM 结构化评分 + 规则校准上线。
2. 推荐策略按短板与风险优先级工作。
3. 证据链可回放。

### P3（2周）：多场景与结局

1. 扩展到 4-6 核心场景。
2. 结局判定与可解释报告完善。
3. 反事实对比可视化。

### P4（1-2周）：验证与答辩

1. 前后测指标验证。
2. 演示脚本稳定化与离线回放。
3. 文档、部署、压测收口。

---

## 10. 验收指标（技术与业务）

1. 评分一致性：与专家评分相关性达到目标阈值。
2. KT 预测力：下一轮表现预测准确率持续提升。
3. 风险错误率：高风险错误（未核实发布、泄露线索）显著下降。
4. 达标效率：达到目标能力所需轮次下降。
5. 可解释性：每轮关键评分至少给出1条证据链。

---

## 11. 风险与缓解

1. LLM 波动风险  
措施：规则校准 + JSON Schema + 回退模板。

2. 历史约束失守风险  
措施：双重约束校验（生成前提示 + 生成后审查）。

3. 冷启动样本不足  
措施：先 BKT，再引入 DKT 校正。

4. 演示稳定性  
措施：预置会话与离线重放机制。

---

## 12. 结论

这套方案可以在不推翻现有仓库的前提下，逐步把系统升级成“历史约束 + 可解释评分 + KT 追踪 + 自适应训练”的完整闭环。  
其关键价值不是“剧情更花哨”，而是“训练结果可量化、可复盘、可持续优化”。
