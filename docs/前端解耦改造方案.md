# 前端解耦改造方案

更新日期：2026-03-18

## 1. 目标

本方案的目标不是重写前端，而是在尽量不打断现有联调节奏的前提下，把当前前端从“页面驱动流程”逐步改成“流程层驱动页面”。

改造完成后，希望达到：

1. 页面组件只负责展示和交互，不直接编排整条业务链路。
2. API 调用、存储读写、路由跳转、错误恢复分别落在独立层。
3. 跨页面流程状态不再主要依赖 `sessionStorage/localStorage` 作为隐式总线。
4. `Game` 页不再同时管理初始化、恢复、提交、场景切换、消息落盘、TTS。
5. 后端接口结构变化时，影响尽量收敛在 `services` 与流程层，而不是扩散到各页面。

## 2. 当前问题摘要

当前前端的主要耦合集中在“页面流程层”，而不是基础组件层。

### 2.1 页面职责过重

以下页面同时承担了 UI、API、存储、导航、错误恢复：

- `frontend/src/pages/CharacterSetting.tsx`
- `frontend/src/pages/CharacterSelection.tsx`
- `frontend/src/pages/FirstMeetingSelection.tsx`
- `frontend/src/pages/FirstStep.tsx`
- `frontend/src/pages/Game.tsx`

### 2.2 跨页面流程依赖存储键

当前角色创建、角色选择、初遇场景、继续游戏、游戏初始化之间，主要靠这些键串联：

- `characterData`
- `createdCharacterId`
- `restoreThreadId`
- `restoreCharacterId`
- `gameThreadId`
- `gameCharacterId`
- `currentCharacterId`
- `initialGameData`
- `gameSave`

这些键定义在：

- `frontend/src/storage/gameStorage.ts`

问题在于：

1. 页面顺序被隐式绑定。
2. 某个字段改名容易产生静默错误。
3. 页面之间不是显式依赖，而是“约定式依赖”。

### 2.3 `services/api.ts` 过胖

当前 `frontend/src/services/api.ts` 同时承担：

1. Axios 实例创建
2. 请求/响应拦截
3. token 处理
4. 业务 API 函数
5. URL 规范化工具 `getStaticAssetUrl`
6. 一部分业务类型定义

这会导致：

1. 服务层边界不清晰
2. UI 组件反向依赖网络服务文件
3. 类型与 API 实现耦在一起

### 2.4 `Game` 流程虽拆 hook，但仍是强耦合

`Game` 页已经拆出：

- `useGameState`
- `useGameInit`
- `useGameTts`

但 `useGameInit` 直接吃整份 `GameStateBag`，说明“页面状态”和“业务行为”仍然是强绑定关系，并没有真正形成稳定的流程模型。

## 3. 改造原则

### 3.1 不推倒重来

保持现有：

- React Router
- Ant Design
- axios
- 当前页面路由结构

优先做“可逐步迁移”的改造，不要求一次性切到新架构。

### 3.2 不先引入新状态库

第一阶段不强依赖 Zustand/Redux。

优先使用：

- `Context + reducer`
- 自定义 flow hooks
- 已有的 `services` / `storage` / `types`

等流程层稳定后，再评估是否需要更强的状态管理库。

### 3.3 保持页面外观不动，先拆职责

优先拆：

- 业务流程
- 数据依赖
- API 边界
- 存储边界

而不是先改样式或目录美化。

## 4. 目标分层

建议把前端职责拆成 5 层：

### 4.1 View 层

职责：

- 页面结构
- UI 渲染
- 事件触发

目录建议：

- `src/pages/*`
- `src/components/*`

要求：

- 不直接读写 `gameStorage`
- 尽量不直接调用 `services/api`
- 不直接拼装复杂业务响应

### 4.2 Flow 层

职责：

- 页面级业务流程编排
- 页面进入时初始化
- 用户操作后的导航判断
- 对 API / storage / view model 的协调

目录建议：

- `src/flows/useCharacterCreationFlow.ts`
- `src/flows/useCharacterSelectionFlow.ts`
- `src/flows/useFirstMeetingFlow.ts`
- `src/flows/useGameSessionFlow.ts`

### 4.3 Domain Service 层

职责：

- 领域 API 封装
- 后端响应解包
- 领域级错误翻译

目录建议：

- `src/services/httpClient.ts`
- `src/services/characterApi.ts`
- `src/services/gameApi.ts`
- `src/services/ttsApi.ts`
- `src/services/healthApi.ts`

### 4.4 Session Store 层

职责：

- 管理“当前流程中的共享状态”
- 统一读写浏览器存储
- 提供内存态与持久化态的桥接

目录建议：

- `src/contexts/GameFlowContext.tsx`
- `src/storage/gameStorage.ts`

要求：

- `gameStorage` 只做持久化适配
- `Context` 才是页面共享状态的主入口

### 4.5 Shared Utility / Contract 层

职责：

- 通用类型
- URL 处理
- 场景映射
- 纯函数工具

目录建议：

- `src/types/*`
- `src/utils/*`
- `src/config/*`

## 5. 目录改造建议

建议在当前结构基础上新增以下文件，不必大搬家：

```text
frontend/src/
  services/
    httpClient.ts
    assetUrl.ts
    healthApi.ts
    characterApi.ts
    gameApi.ts
    ttsApi.ts
  flows/
    useCharacterCreationFlow.ts
    useCharacterSelectionFlow.ts
    useFirstMeetingFlow.ts
    useGameSessionFlow.ts
  contexts/
    GameFlowContext.tsx
    index.ts
```

保留：

- `pages`
- `components`
- `hooks`
- `storage`
- `types`
- `config`

说明：

- 第一阶段不强制把现有 `hooks` 改名。
- 可以先新增 `flows`，再逐步把页面里的流程代码迁过去。

## 6. 分阶段执行方案

## 6.1 第一阶段：服务层拆分

目标：先把最胖的 `services/api.ts` 拆薄，但不改页面行为。

### 要做什么

1. 新增 `httpClient.ts`
2. 把 axios 实例、拦截器、token 逻辑放进去
3. 新增 `assetUrl.ts`
4. 把 `getStaticAssetUrl` 从 `api.ts` 挪出去
5. 按领域拆分 API：
   - `characterApi.ts`
   - `gameApi.ts`
   - `ttsApi.ts`
   - `healthApi.ts`
6. `types/api.ts` 成为 API 契约类型的单一来源

### 对现有文件的影响

- `frontend/src/services/api.ts`
- `frontend/src/types/api.ts`
- `frontend/src/components/Game/GameSceneBackground.tsx`
- `frontend/src/pages/*`
- `frontend/src/hooks/useGameInit.ts`
- `frontend/src/hooks/useGameTts.ts`

### 这一阶段的收益

1. UI 不再依赖一个巨型服务文件
2. 后端接口变更时影响面更集中
3. 类型漂移风险下降

### 风险

- 低

这是最适合优先做的一步。

## 6.2 第二阶段：把浏览器存储从“流程总线”降级为“持久化适配”

目标：减少页面之间对 `sessionStorage/localStorage` 键名的直接耦合。

### 要做什么

1. 新增 `GameFlowContext.tsx`
2. 在 Context 中维护当前流程共享数据：
   - `characterDraft`
   - `selectedCharacter`
   - `selectedScene`
   - `voiceConfig`
   - `runtimeSession`
3. `gameStorage.ts` 保留，但只提供：
   - 加载初始持久化数据
   - 持久化保存
   - 页面刷新恢复
4. 页面改为优先读 Context，而不是到处直接读 `gameStorage`

### 推荐状态模型

```ts
type GameFlowState = {
  characterDraft: CharacterData | null;
  selectedScene: SelectedScene | null;
  runtimeSession: {
    threadId: string | null;
    characterId: string | null;
    initialGameData: InitialGameData | null;
  };
};
```

### 对现有文件的影响

- `frontend/src/storage/gameStorage.ts`
- `frontend/src/pages/CharacterSetting.tsx`
- `frontend/src/pages/CharacterSelection.tsx`
- `frontend/src/pages/FirstMeetingSelection.tsx`
- `frontend/src/pages/FirstStep.tsx`
- `frontend/src/hooks/useGameInit.ts`
- `frontend/src/contexts/index.ts`

### 这一阶段的收益

1. 页面之间的数据依赖变显式
2. 键名变化不再传播到页面层
3. 更容易调试和测试流程状态

### 风险

- 中

因为它会触及多个页面，但收益很大。

## 6.3 第三阶段：页面流程迁移到 flow hooks

目标：让页面从“业务 orchestrator”变成“视图 + 事件入口”。

### 6.3.1 `CharacterSetting`

新增：

- `useCharacterCreationFlow.ts`

从页面迁出的内容：

- 健康检查
- 角色请求体组装
- 创建角色
- 响应校验
- 存储写入
- 跳转逻辑

页面保留：

- 表单状态
- UI 展示
- 点击确认时调用 `createCharacterFlow.submit()`

### 6.3.2 `CharacterSelection`

新增：

- `useCharacterSelectionFlow.ts`

从页面迁出的内容：

- 角色加载
- 图片选择后的状态更新
- 去背请求
- 音色列表拉取
- 试听
- 音色保存
- 成功后跳转

页面保留：

- 图片选择界面
- 音色选择界面
- 按钮与展示逻辑

### 6.3.3 `FirstMeetingSelection`

新增：

- `useFirstMeetingFlow.ts`

从页面迁出的内容：

- 场景列表加载
- 选择场景后初始化游戏
- 初始化故事
- 初始数据写入 runtime/session store
- 跳转 `Game`

页面保留：

- 场景轮播
- 选择按钮

### 6.3.4 `Game`

新增：

- `useGameSessionFlow.ts`

整合：

- `useGameInit`
- `useGameTts`
- 当前 `Game.tsx` 里的提交/恢复逻辑

建议把 `Game` 页最终收口为：

1. 读取 `flow`
2. 渲染背景与对话
3. 调用 `flow.selectOption(index)`

## 6.4 第四阶段：收缩 `GameStateBag`

目标：避免 hook 和页面之间通过一大堆 setter/ref 紧耦合。

### 当前问题

`useGameState` 暴露了大量内部细节：

- `setMessages`
- `setLoading`
- `setCurrentScene`
- `setCharacterImageUrl`
- `previousSceneRef`
- 大量其他 setter

这导致：

1. `useGameInit` 知道过多内部状态
2. `Game.tsx` 知道过多流程细节
3. 状态边界模糊

### 改造方向

把 `GameStateBag` 拆成：

1. `state`
2. `actions`
3. `derived`

示例：

```ts
type GameSessionState = {
  threadId: string | null;
  characterId: string | null;
  messages: GameMessage[];
  currentDialogue: string;
  currentOptions: PlayerOption[];
  currentScene: string | null;
  sceneImageUrl: string | null;
  characterImageUrl: string | null;
  compositeImageUrl: string | null;
  loading: boolean;
};

type GameSessionActions = {
  initialize(): Promise<void>;
  selectOption(index: number): Promise<void>;
  recoverSession(): Promise<void>;
};
```

让页面只能调动作，而不是任意改内部状态。

## 6.5 第五阶段：减少前后端场景命名耦合

目标：减少前端对后端静态资源命名规则的猜测。

### 当前问题

以下逻辑是高风险耦合点：

- `frontend/src/config/scenes.ts`
- `frontend/src/utils/game.ts`

尤其：

- 前端在猜测静态资源文件名
- 前端自己维护一份场景中文名和场景 ID 映射

### 建议

优先顺序：

1. 以后端接口返回的 `scene_image_url` / `imageUrl` 为主
2. 前端 fallback 只保留最小兜底
3. `SCENE_CONFIGS` 逐步降级成“展示配置”，不要再承担资源路径协议

## 7. 文件级拆分建议

下面是一个低风险拆法。

### 第一步先新增，不删旧代码

新增：

- `src/services/httpClient.ts`
- `src/services/assetUrl.ts`
- `src/services/characterApi.ts`
- `src/services/gameApi.ts`
- `src/services/ttsApi.ts`

然后让旧 `api.ts` 仅做 re-export，先保证引用不炸。

### 第二步逐页迁移

迁移顺序建议：

1. `Home.tsx`
2. `FirstStep.tsx`
3. `CharacterSetting.tsx`
4. `CharacterSelection.tsx`
5. `FirstMeetingSelection.tsx`
6. `Game.tsx`

原因：

- 前两页逻辑轻，适合先建立 flow 模式
- 中间三页是创建链路
- `Game` 最复杂，应该最后处理

### 第三步再清理旧入口

当页面都迁移完成后：

1. 删除或瘦身旧 `api.ts`
2. 删除不再需要的存储直接访问
3. 清理重复类型定义

## 8. 优先级清单

### 必须先做

1. 拆 `services/api.ts`
2. 把 `getStaticAssetUrl` 挪出服务层
3. 把 `CharacterSelection` 的流程逻辑抽出去
4. 把 `FirstMeetingSelection` 的初始化流程抽出去
5. 收缩 `Game` 页中的恢复与提交逻辑

### 建议第二批做

1. 引入 `GameFlowContext`
2. 缩小 `GameStateBag`
3. 统一 API 类型来源
4. 清理 `utils/game.ts` 中对静态资源命名的硬编码猜测

### 可后置

1. 是否引入 Zustand
2. 页面内部进一步拆分小组件
3. 样式文件整理

## 9. 预计工作量

如果按低风险增量方式推进，建议这样估时：

### 第 1 批

- 服务层拆分
- 类型统一
- 资源 URL 工具拆出

预计：0.5 ~ 1 天

### 第 2 批

- `CharacterSetting`
- `CharacterSelection`
- `FirstMeetingSelection`

预计：1.5 ~ 2 天

### 第 3 批

- `Game` 页收口
- `useGameInit / useGameTts / useGameState` 重整

预计：1.5 ~ 2.5 天

### 第 4 批

- Context 引入
- 存储角色降级
- 清理旧逻辑

预计：1 ~ 1.5 天

总计：

- 大约 4.5 ~ 7 天

## 10. 验收标准

完成解耦后，至少应满足：

1. 页面组件中不再直接出现大段 API 编排代码。
2. `gameStorage` 不再是跨页面流程主入口。
3. `GameSceneBackground` 不再依赖 `services/api.ts`。
4. `Game.tsx` 中不再直接承担会话恢复主逻辑。
5. `CharacterSelection.tsx` 不再同时处理图片选择、TTS 拉取、试听、去背、音色写回。
6. `services`、`types`、`storage`、`flow` 之间边界清晰。

## 11. 推荐的实际落地顺序

最推荐的推进顺序如下：

1. 先拆 `services/api.ts`
2. 再改 `CharacterSelection.tsx`
3. 再改 `FirstMeetingSelection.tsx`
4. 再改 `Game.tsx`
5. 最后引入 `GameFlowContext` 收口流程状态

原因：

1. 风险最小
2. 每一步都能看到明显收益
3. 不会一开始就碰最复杂的共享状态改造
