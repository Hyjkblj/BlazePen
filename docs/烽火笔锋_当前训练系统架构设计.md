# 烽火笔锋_当前训练系统架构设计

## 1. 当前版本定位

当前版本处于二次开发的 **P1/P2 稳定落地阶段**，目标不是把整套训练系统一次性做成“大而全平台”，而是先把训练主链路做成：

1. 可初始化训练会话
2. 可按场景顺序推进回合
3. 可对用户输入做规则评估 / LLM 评估
4. 可更新 K 状态与 S 状态
5. 可生成结局与训练报告
6. 可持久化完整回放数据

当前版本 **不依赖 Chroma**，重点先放在：

1. 训练链路稳定
2. 数据结构清晰
3. 配置可维护
4. 代码低耦合
5. 后续便于扩展到题库、候选推荐、复盘检索

---

## 2. 当前总体架构

当前训练系统可以理解为 8 层：

1. `API / Service 编排层`
2. `Policy 策略层`
3. `Evaluator 评估层`
4. `Scenario Repository 场景库层`
5. `Contract 契约层`
6. `Store / Repository 持久化层`
7. `Output DTO 输出层`
8. `Database Session Factory 连接层`
9. `Runtime Config 配置层`

整体关系如下：

```text
前端 / 路由
    ↓
TrainingService
    ├─ TrainingModeCatalog
    ├─ ScenarioPolicy
    ├─ ScenarioRepository
    ├─ RecommendationPolicy
    ├─ TrainingTelemetryPolicy
    ├─ TrainingRoundEvaluator
    ├─ EndingPolicy
    ├─ TrainingOutput DTOs
    └─ TrainingStore
            └─ SqlAlchemyTrainingRepository
                    ├─ TrainingSession
                    ├─ TrainingRound
                    ├─ RoundEvaluation
                    ├─ KtStateSnapshot
                    ├─ NarrativeStateSnapshot
                    ├─ ScenarioRecommendationLog
                    ├─ TrainingAuditEvent
                    ├─ KtObservation
                    └─ EndingResult

DatabaseManager
    ├─ Character
    ├─ CharacterAttribute
    ├─ CharacterState
    └─ 训练仓储兼容转发层（仅保留旧调用兼容）

database.session_factory
    ├─ get_engine()
    ├─ get_session_factory()
    └─ build_database_url()

Runtime Config(JSON + Loader)
    ├─ 场景序列
    ├─ 默认 K/S 状态
    ├─ 规则词表
    ├─ 奖惩参数
    ├─ 结局阈值
    └─ LLM 运行参数

Scenario Bank(JSON + Repository)
    ├─ 场景正文
    ├─ 任务目标
    ├─ 目标能力
    ├─ 风险标签
    └─ 预设选项
```

## 3. 分层职责

### 3.1 Service 编排层

核心文件：

1. `backend/api/services/training_service.py`

职责：

1. 初始化训练会话
2. 串联“场景校验 -> 回合评估 -> 状态更新 -> 持久化 -> 结局生成”
3. 提供 `init_training / get_next_scenario / submit_round / get_progress / get_report`
4. 处理幂等回包
5. 统一规范化训练模式输入，避免别名和脏值散落
6. 对外返回稳定的数据结构，不把内部临时字段直接暴露给前端
7. 只处理领域语义，不直接理解数据库驱动异常

这一层现在的原则是：**只编排，不存业务规则细节**。

### 3.2 Policy 策略层

核心文件：

1. `backend/training/scenario_policy.py`
2. `backend/training/ending_policy.py`
3. `backend/training/recommendation_policy.py`
4. `backend/training/telemetry_policy.py`

职责：

1. `ScenarioPolicy`
   负责冻结场景顺序、恢复会话场景序列、校验用户是否按顺序提交
2. `EndingPolicy`
   负责根据最终 K/S 状态与风险历史，生成结局类型和结局说明
3. `RecommendationPolicy`
   负责在推荐模式下，根据当前 K 短板和 S 状态风险完成下一题排序
4. `TrainingTelemetryPolicy`
   负责把推荐日志、审计事件、KT 结构化观测从服务层收口成稳定工件

补充工具：

1. `backend/training/training_mode.py`
   负责训练模式 canonical 化、别名收敛和模式能力判定

这一层把“流程策略”“结局策略”“训练观测策略”从 Service 中拆开，避免 `TrainingService` 继续膨胀。

### 3.3 Evaluator 评估层

核心文件：

1. `backend/training/evaluator.py`

职责：

1. 先执行规则评估
2. 可选执行 LLM 评估
3. 融合规则结果与 LLM 结果
4. 对高风险标签施加硬惩罚
5. 输出统一评估结构

这一层当前已经改成：

1. 算法逻辑保留在代码里
2. 业务参数来自外部 JSON 配置

也就是说，现在代码只管“怎么算”，配置只管“算什么”。

### 3.4 场景库层

核心文件：

1. `backend/training/config/scenario_bank.json`
2. `backend/training/scenario_repository.py`

职责：

1. 维护完整场景定义
2. 负责按场景 ID 查询完整内容
3. 在训练初始化时冻结场景快照
4. 避免 `TrainingService` 直接硬编码场景正文

这一层解决的是过去最容易继续膨胀的问题：

1. 场景配置混在 service 里
2. 只能返回 `id/title`
3. 场景发版后老会话内容可能漂移

现在训练会话初始化时，会把完整场景快照冻结到 `session_meta`，老会话优先读自己的冻结版本。

### 3.5 Contract 契约层

核心文件：

1. `backend/training/contracts.py`

职责：

1. 统一评估结果结构
2. 统一推荐日志、审计事件、KT 观测等内部工件结构
3. 对 `confidence / risk_flags / skill_delta / s_delta / evidence` 等字段做归一化
4. 让评估层、服务层、存储层看到同一套 payload 契约

这一层的价值是：**减少 dict 键名散落和下游判断分支**。

### 3.6 持久化层

核心文件：

1. `backend/training/training_store.py`
2. `backend/training/training_repository.py`
3. `backend/database/db_manager.py`
4. `backend/database/session_factory.py`
5. `backend/database/integrity.py`

职责：

1. `TrainingStore`
   对服务层暴露最小训练持久化接口
2. `SqlAlchemyTrainingRepository`
   训练域专用真实仓储，承担事务落库和训练表访问
3. `session_factory`
   统一管理 engine / sessionmaker，避免角色域和训练域各自建连接池
4. `integrity.py`
   统一识别唯一约束冲突，减少驱动异常文本判断散落
5. `DatabaseManager`
   现在主要负责角色域；训练相关方法只保留兼容转发，不再直接持有训练契约实现
6. `TrainingStore`
   把底层 ORM / 行对象转换成稳定记录模型，再交给服务层使用
7. 回合、评估、快照、推荐日志、审计事件、KT 观测、结局的原子化落库
8. 回放查询与报告查询

当前设计里，服务层已经不再直接依赖 `DatabaseManager`，而是先经过 `TrainingStore` 适配层，再落到底层数据库实现。

进一步说，现在训练域已经变成：

1. `TrainingService -> TrainingStore -> SqlAlchemyTrainingRepository`
2. `DatabaseManager` 不再是训练服务默认入口
3. 旧代码如果还在调 `DatabaseManager.create_training_*`，仍能通过兼容转发继续工作

### 3.7 Output DTO 输出层

核心文件：

1. `backend/training/training_outputs.py`

职责：

1. 定义训练服务对外返回的稳定输出模型
2. 收口初始化、下一题、提交结果、进度、报告、诊断摘要的字典导出
3. 避免 `TrainingService` 内部散落大段响应拼装逻辑

这一层的价值是：**让服务编排和响应组装分开演进**。

### 3.8 配置层

核心文件：

1. `backend/training/config/training_runtime_config.json`
2. `backend/training/config_loader.py`
3. `backend/training/constants.py`

职责：

1. JSON 文件保存业务规则参数
2. Loader 负责加载、校验、缓存、环境变量覆盖
3. `constants.py` 只做统一导出，不再堆积业务硬编码

## 4. 当前请求链路

### 4.1 初始化训练

链路：

1. `TrainingService.init_training`
2. `TrainingModeCatalog` 规范化训练模式
3. `ScenarioRepository` 把摘要场景序列补齐成完整场景快照
4. `ScenarioPolicy` 冻结当前场景序列与完整场景快照
5. `TrainingTelemetryPolicy` 生成 `session_initialized` 审计事件
6. `TrainingStore.create_training_session_artifacts`
7. 原子写入 round=0 的 K/S 快照与初始化审计事件

结果：

1. 得到会话 ID
2. 默认 K/S 状态写入数据库
3. 场景版本、场景序列、场景快照写入 `session_meta`
4. 若为 `self-paced`，会同时返回推荐第一题和候选题列表

### 4.2 提交单回合

链路：

1. `TrainingService.submit_round`
2. 读取会话当前状态
3. 读取会话冻结的场景快照
4. 若为推荐模式，`RecommendationPolicy` 计算剩余场景排序
5. `adaptive` 严格校验第一推荐题，`self-paced` 校验是否命中候选题集合
6. 基于同一份候选结果构建 `decision_context`
7. `TrainingRoundEvaluator.evaluate_round`
8. 更新 K 状态与 S 状态
9. 若到最后一轮，`EndingPolicy.resolve`
10. `TrainingTelemetryPolicy` 生成推荐日志、审计事件、KT 观测
11. `TrainingStore.save_training_round_artifacts`
12. 返回本回合结果

这一条链路里最重要的设计点有 5 个：

1. 评估结果先过 `RoundEvaluationPayload`
2. 结局与回合记录同事务提交
3. 推荐决策上下文会结构化落到 `training_rounds.user_action.decision_context`
4. 推荐日志、审计事件、KT 观测会与回合工件同事务提交
5. 重复提交优先返回幂等结果，而不是直接报错

### 4.3 API 契约稳定化

这一轮之后，训练接口对外返回开始收口成明确 schema，而不再长期停留在 `response_model=dict`：

1. 初始化接口返回 `next_scenario` 与可选 `scenario_candidates`
2. 下一场景接口无论是否完成，都返回稳定字段：
   `session_id / status / round_no / scenario / scenario_candidates / k_state / s_state / ending`
3. 进度接口和报告接口也有了明确响应模型

这样做的价值是：

1. 前端不用在“进行中 / 已完成”两种状态下写两套解析逻辑
2. OpenAPI 文档更准确
3. 后续再扩字段时，变更边界更清晰
4. 服务层内部也能通过 DTO 统一输出结构，而不是继续手写 dict

### 4.4 查询报告

链路：

1. `TrainingService.get_report`
2. 读取训练回合历史
3. 读取每回合评估
4. 组装回放数据
5. 计算最终提升值 `improvement`

输出：

1. 最终 K 状态
2. 最终 S 状态
3. 结局
4. 每一轮的输入、评估、前后状态变化
5. 每一轮提交时的 `decision_context`，用于复盘“系统当时推荐了什么、用户最终选了什么”
6. 每一轮对应的 `kt_observation`，用于直接查看“本轮重点补了什么、触发了什么风险、能力怎么变化”

### 4.5 训练诊断查询

链路：

1. `TrainingService.get_diagnostics`
2. `TrainingStore.get_scenario_recommendation_logs`
3. `TrainingStore.get_training_audit_events`
4. `TrainingStore.get_kt_observations`
5. 输出稳定诊断 DTO

输出：

1. `recommendation_logs`
   还原每轮推荐候选、推荐题和用户最终选择
2. `audit_events`
   还原初始化、提交、完成等关键流程事件
3. `kt_observations`
   还原每轮训练事实，便于做诊断面板和报告可视化
4. `summary`
   聚合高风险轮次、推荐偏差、主技能关注点、弱项统计和事件计数，减少前端重复计算

## 5. 当前算法架构

### 5.1 状态模型

系统内部维护两类状态：

1. `K 状态`
   对应能力成长维度，例如核验、来源保护、纠错等
2. `S 状态`
   对应剧情运行状态，例如可信度、准确性、公众恐慌、线人安全等

### 5.2 回合评估算法

当前一轮评估流程：

1. 规则评估器扫描关键词规则
2. 命中规则后，对 `skill_delta` 和 `s_delta` 做加减
3. 如果启用 LLM，则让 LLM 输出结构化评估 JSON
4. 把 LLM 输出和规则输出按权重融合
5. 如果命中高风险标签，再追加硬惩罚
6. 最终结果统一做限幅、归一化、补默认值

### 5.3 结局判定算法

当前结局判定来自三类信息：

1. 最终综合 K 分数
2. 最终 S 状态区间
3. 历史风险标记

判定顺序：

1. 先看是否触发严重风险
2. 再看是否达到优秀结局阈值
3. 再看是否达到恢复型结局阈值
4. 再看是否达到稳健型结局阈值
5. 否则进入失败型结局

这一顺序的目的很明确：**不能因为“最后总分高”就把严重伦理风险洗掉**。

### 5.4 下一题推荐算法

当前已经支持 `adaptive` 和 `self-paced` 两种推荐模式，推荐逻辑现在来自四部分：

1. `K` 短板覆盖值
2. `S` 状态触发加权
3. 最近风险历史补救加权
4. 当前回合阶段对齐加权

当前计算方式是：

1. 先看一个场景覆盖了哪些目标能力
2. 再计算这些能力当前有多弱
3. 如果某些 `S` 状态触发风险阈值，就给相关技能场景额外加分
4. 如果最近几轮反复出现某类风险标记，就给对应补救型场景额外加分
5. 再根据当前回合位置，给与当前训练阶段更贴近的题目补一层阶段分
6. 对已完成场景直接排除
7. 把剩余场景按总分排序
8. `adaptive` 取第一名作为唯一允许提交题目
9. `self-paced` 取第一名作为推荐题，再返回前 N 个候选场景给用户自由选择

这里新增的两类信号分别解决两个老问题：

1. 只看 K/S 静态状态时，系统不知道“你刚刚连续犯了什么错”
2. 只看能力短板时，推荐可能在叙事节奏上跳得太远

所以现在推荐分数可以理解成：

1. 基础分：这题能不能补你当前最弱的能力
2. 状态分：这题能不能缓解当前剧情状态风险
3. 风险分：这题能不能针对你最近反复出现的错误做补救
4. 阶段分：这题跟当前训练进度是否贴近

当前三种模式分工如下：

1. `guided`
   固定顺序推进
2. `adaptive`
   严格命中推荐第一题
3. `self-paced`
   返回推荐排序，但允许用户在候选列表内自由选题

## 7. 当前数据库落点

当前训练链路主要围绕以下表：

1. `training_sessions`
   保存训练会话主状态
2. `training_rounds`
   保存每回合用户输入与前后状态
3. `round_evaluations`
   保存每回合评估详情
4. `kt_state_snapshots`
   保存每回合 K 状态快照
5. `narrative_state_snapshots`
   保存每回合 S 状态快照
6. `scenario_recommendation_logs`
   保存每回合推荐候选、推荐题与最终选择
7. `training_audit_events`
   保存训练初始化、提交、完成等关键事件
8. `kt_observations`
   保存每回合结构化训练观测，便于诊断与报表复用
9. `ending_results`
   保存最终结局
10. `alembic_version`
   保存当前数据库迁移版本，确保 schema 变更有明确落点
11. `users / threads / story_states / conversations / image_cache`
   旧无限剧情流核心表，当前也已纳入迁移管理，避免数据库只托管训练域一半结构

当前数据库设计的特点：

1. 主状态在 `training_sessions`
2. 每轮过程在 `training_rounds`
3. 决策回放上下文落在 `training_rounds.user_action.decision_context`
4. 推荐链路另存 `scenario_recommendation_logs`，避免后续分析反复解析 `user_action`
5. 关键流程事件另存 `training_audit_events`，便于排查、治理与回放
6. 每轮训练事实另存 `kt_observations`，便于做诊断和报表
7. 结局独立存储，便于报告直接读取
8. 数据库初始化与升级开始收口到 Alembic 迁移，应用启动时不再自动补表
9. Alembic `20260317_0002` 已把旧剧情流核心表也纳入统一治理

## 8. 当前低耦合设计点

这一轮之后，耦合度下降最明显的地方有 12 个：

1. `TrainingService` 不再依赖数据库驱动异常文本
2. 评估层、服务层、存储层共用 `RoundEvaluationPayload`
3. 场景策略、结局策略已拆出独立 Policy
4. 业务规则参数已从代码迁移到外部 JSON
5. 场景正文已从 Service 中迁移到独立场景库

## 10. 一句话总结

当前这套架构已经从“把训练逻辑塞进 service 和常量文件”升级成了：

**Service 负责编排，ModeCatalog 负责模式规范化，Policy 负责策略，TelemetryPolicy 负责观测工件，Evaluator 负责算法，Contract 负责评估与内部工件契约，Output DTO 负责对外结果结构，TrainingStore/TrainingRepository 负责训练持久化，DatabaseManager 主要负责角色域兼容访问，session_factory 负责数据库连接，JSON Config 负责业务参数。**

## 16. 本轮补充（2026-03-17，阶段化推荐配置）

本轮继续沿着“低耦合、可插拔、配置化”的方向，把训练引擎里的“剧情节奏推荐”也正式拆成了配置驱动：

1. `scenario_bank.json`
   每个场景新增 `phase_tags`，用于声明自己属于哪个剧情阶段。
   当前默认使用：
   `opening / middle / climax / closing`
2. `training_runtime_config.json`
   新增 `flow.stage_windows`，用来声明“第几轮处于哪个训练阶段”。
3. `RecommendationPolicy`
   现在会先解析当前轮次命中的阶段窗口，再用阶段标签去匹配场景，而不是只靠场景顺序做推荐。

当前阶段推荐的计算逻辑已经变成：

1. 先算 K 短板覆盖值
2. 再算 S 状态触发加权
3. 再算最近风险补救加权
4. 最后算阶段加权：
   如果当前轮命中 `stage_windows`，且场景本身带有 `phase_tags`，优先走“阶段标签匹配”
   如果有多个场景同时命中同一阶段，再用原有距离加权细排
   如果场景没有阶段标签，则完全回退到旧的距离加权逻辑

这一步解决的是一个很典型的耦合问题：

1. 过去的“第几轮更适合推什么题”主要绑在场景原始顺序上
2. 现在的“剧情阶段”与“场景顺序”被拆成了两个概念
3. 场景可以靠 `phase_tags` 描述自己
4. 训练流程可以靠 `stage_windows` 描述当前节奏
5. 推荐策略只负责把两者匹配起来

这样做的直接收益有 4 个：

1. 后续扩场景时，节奏控制主要改配置，不需要频繁改推荐代码
2. 场景顺序和剧情节奏不再是强绑定关系，推荐更灵活
3. 老数据或临时场景即使还没补 `phase_tags`，系统也不会失效，因为还有旧距离算法兜底
4. 后面如果要为 `guided / adaptive / self-paced` 配不同阶段节奏，也已经有了明确挂点

## 18. 本轮补充（2026-03-17，阶段诊断汇总）
