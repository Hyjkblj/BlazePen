## 2. 用户视角的完整交互路径

### 2.1 进入训练

1. 用户先进入 `/training/mainhome`
2. 点击进入训练后跳转到 `/training/landing`
3. 在 landing 页，系统判断是否存在可恢复训练入口
4. 如果没有可恢复会话，用户填写训练模式、角色和玩家档案后开始训练
5. 如果有可恢复会话，用户可以直接恢复最近训练

### 2.2 开始新训练

1. 前端调用 `POST /api/v1/training/init`
2. 后端返回新的 `session_id`、初始 `runtime_state` 和 `next_scenario`
3. 前端写入内存态 `activeSession`
4. 前端同时写入本地 `trainingResumeTarget`
5. 页面进入 `/training` 主训练页，展示当前场景和提交区

### 2.3 恢复已有训练

1. 前端调用 `GET /api/v1/training/sessions/{session_id}`
2. 后端返回训练恢复摘要
3. 前端据此重建 `sessionView`
4. 如果后端返回 `TRAINING_SESSION_NOT_FOUND` 或 `TRAINING_SESSION_RECOVERY_STATE_CORRUPTED`
5. 前端会清理本地恢复入口，避免继续拿脏会话做恢复

### 2.4 提交训练回合

1. 用户在训练页选择一个选项，或者输入本轮行动说明
2. 前端调用 `POST /api/v1/training/round/submit`
3. 如果提交成功且训练未完成，前端继续调用 `POST /api/v1/training/scenario/next`
4. 如果下一场景成功返回，页面切换到新的训练场景
5. 如果下一场景加载失败，前端会退回到 `restoreSession`，按服务端当前摘要恢复

### 2.5 回合异常时的恢复策略

下列错误不会简单停在报错，而是优先尝试恢复到服务端权威状态：

- `TRAINING_ROUND_DUPLICATE`
- `TRAINING_SESSION_COMPLETED`
- `TRAINING_SCENARIO_MISMATCH`
- `TRAINING_SESSION_NOT_FOUND`
- `TRAINING_SESSION_RECOVERY_STATE_CORRUPTED`

恢复成功后，页面按服务端 summary 重建状态，并显示对应 notice。

### 2.6 训练完成后的查看路径

训练完成后，用户可以继续从页头进入三个只读洞察页：

- `/training/progress?sessionId=...`
- `/training/report?sessionId=...`
- `/training/diagnostics?sessionId=...`

这三个页面分别读取：

- `GET /api/v1/training/progress/{session_id}`
- `GET /api/v1/training/report/{session_id}`
- `GET /api/v1/training/diagnostics/{session_id}`

## 3. 当前关键字段

训练用户交互流程里最关键的字段如下：

| 字段 | 含义 |
| --- | --- |
| `session_id` | 训练主线唯一会话主键 |
| `training_mode` | 训练模式，canonical 值为 `guided / self-paced / adaptive` |
| `scenario_id` | 训练场景主键 |
| `round_no` | 当前回合号 |
| `runtime_state` | 当前运行时状态总视图 |
| `decision_context` | 本轮决策上下文 |
| `consequence_events` | 本轮后果事件 |
| `progress_anchor` | 训练恢复与进度锚点 |
| `is_completed` | 训练是否完成 |

## 4. 设计上的边界说明

1. `trainingResumeTarget` 只是前端 UX 恢复缓存，不是服务端事实源。
2. 真正恢复训练必须走 `GET /api/v1/training/sessions/{session_id}`。
3. 训练主线只使用 `session_id`，不能混入故事主线的 `thread_id` 概念。
4. 页面不应自行拼装训练报告或诊断数据，而应消费后端返回的只读聚合模型。
5. 提交失败后的优先动作不是本地猜测，而是回到服务端恢复摘要。
