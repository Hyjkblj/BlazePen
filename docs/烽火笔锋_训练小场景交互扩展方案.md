# 烽火笔锋训练后端小场景交互扩展方案

## 任务理解
- 当前训练主线虽然已经有 `S1-S6` 六个大场景，并存在 `S2B`、`S3R` 这类少量补救支线，但玩家主观感受仍然接近“只做 6 次关键选择”，互动密度偏低。
- 目标不是把训练改成无限随机故事，而是在不偏离抗战记者训练主线的前提下，在大场景衔接阶段插入 `2-3` 个可交互小场景，增加决策频次、过程张力和状态反馈。
- 小场景必须是训练域的一等公民，而不是前端临时拼出来的过场文本；它们需要有明确 `sceneId`、持久化、状态影响、恢复依据和回放结果。
- 大场景仍然是历史主锚点和结局主线，小场景只负责“主线之间的过程决策”和“局部状态放大/修复”，不能抢走大场景的历史叙事职责。
- 随机不等于放飞。小场景的随机应是“受历史窗口和主线阶段约束的随机抽取”，而不是在运行时任意生成不受控内容。
- 本方案只讨论后端训练域设计，不扩散到 story 域，不要求前端先承担编排责任。

## 后端方案

### router
- 训练路由保持现有职责，只消费统一的训练场景 DTO。
- 路由层不新增“小场景专用接口”；继续复用：
  - `POST /api/v1/training/init`
  - `POST /api/v1/training/scenario/next`
  - `POST /api/v1/training/round/submit`
  - `GET /api/v1/training/progress/{session_id}`
  - `GET /api/v1/training/report/{session_id}`
  - `GET /api/v1/training/diagnostics/{session_id}`
- 路由层只返回“当前下一题是什么”，不负责推断这是大场景还是小场景。

### service
- 保留 `TrainingService` 作为训练会话编排入口，但不继续把“小场景计划生成 + 随机选择 + 恢复 + DTO 拼装”全部塞进同一个 God service。
- 建议新增独立编排组件：
  - `training/transition_scene_service.py`
    - 负责在两个大场景之间生成并冻结本次会话的小场景计划。
  - `training/transition_scene_plan_policy.py`
    - 负责“根据历史窗口 + 当前状态 + 风险标签”筛选 2-3 个小场景。
  - `training/scene_timeline_policy.py`
    - 负责统一决定“下一题返回大场景、小场景还是补救支线”。
- `TrainingService` 只做以下工作：
  - 初始化训练会话时冻结主线大场景序列。
  - 当某个大场景完成后，请求 `transition_scene_service` 生成当前过渡窗口的小场景队列。
  - 提交回合时，根据当前场景类型调用统一的状态迁移逻辑。
  - 报告与诊断读取时，从同一事实源还原“大场景 + 小场景 + 支线”的完整历史。

### repository-store
- 不建议长期把“小场景计划”只塞在进程内或临时变量里。
- 建议新增持久化实体，至少做到“会话启动后冻结，恢复时可读回”。
- 推荐新增两张表：
  - `training_transition_plans`
    - 字段建议：
      - `plan_id`
      - `session_id`
      - `window_id`
      - `from_major_scene_id`
      - `to_major_scene_id`
      - `plan_status`
      - `selection_seed`
      - `selected_count`
      - `created_at`
      - `completed_at`
  - `training_transition_plan_items`
    - 字段建议：
      - `id`
      - `plan_id`
      - `session_id`
      - `scene_id`
      - `scene_level`
      - `scene_order`
      - `parent_major_scene_id`
      - `window_id`
      - `selection_reason`
      - `is_completed`
      - `completed_round_no`
- 如果短期内不想先加表，只能作为过渡兼容方案把计划冻结到 `training_sessions.session_meta.transition_scene_plan` 中。
  - 该方案必须明确标注为过渡兼容，不应成为长期方案。

### policy
- 新增 `transition_scene_plan_policy`
  - 输入：
    - 当前 `session_id`
    - `from_major_scene_id`
    - `to_major_scene_id`
    - 当前 `k_state / s_state`
    - 最近风险标记
    - 当前运行时 `runtime_flags`
  - 输出：
    - 一个冻结后的过渡小场景列表，长度为 `2-3`
- 新增 `transition_scene_guard_policy`
  - 负责约束：
    - 小场景必须属于指定历史窗口
    - 小场景不能越过当前大场景目标
    - 小场景不能改写主线历史锚点
    - 小场景不能把会话跳转到非法主线之外
- 现有 `round_flow_policy` 不直接被推倒重做，而是演进为：
  - 先检查当前是否存在未完成的小场景队列
  - 再检查是否命中失败/补救支线
  - 最后才回到主线大场景选择

### async-task
- 小场景的“结构选择”不应依赖同步 LLM 实时生成。
- 推荐做法：
  - 后端先从受控配置池中随机选择小场景模板
  - 如需更生动的叙事文案，可在异步或弱依赖层做文本扩写
  - 但扩写只允许修改说明性文本，不允许改：
    - `scene_id`
    - `options`
    - `风险标签`
    - `状态增减规则`
    - `next_rules`

### dto
- 现有 `TrainingScenarioResponse` 建议扩展稳定字段：
  - `scene_level`
    - `major | micro | branch`
  - `scene_path_type`
    - `mainline | transition_micro | failure_branch | recovery_branch`
  - `parent_major_scene_id`
  - `transition_window_id`
  - `scene_order_in_window`
  - `window_total_scenes`
- 现有历史/报告项建议补齐：
  - `scene_level`
  - `parent_major_scene_id`
  - `transition_window_id`
  - `state_effect_summary`
- 错误结构不新增 message 语义猜测，继续复用标准错误码与 `details`。

## 一致性与契约设计

### 1. 会话事实源
- 单一事实源在服务端数据库，不在前端缓存，也不在 CLI 内存。
- 会话初始化后，必须冻结：
  - 主线大场景序列
  - 每个大场景间的过渡窗口定义
  - 当前已选中的小场景计划
- 恢复时只读持久化结果，不重新随机。

### 2. 小场景的随机策略
- “随机拟定 2-3 个小场景”应改写为：
  - “从当前历史窗口允许的候选池中，按规则随机抽取 2-3 个小场景并冻结”
- 随机选择输入建议：
  - 历史窗口
  - 当前大场景完成结果
  - 最近风险标签
  - 当前 `k_state / s_state`
  - 固定的 `selection_seed`
- 这样可以保证：
  - 新会话之间有变化
  - 同一会话恢复后不漂移
  - 报告和诊断能回放“当时为什么抽到这几个小场景”

### 3. 恢复策略
- 恢复依据必须包括：
  - `session_id`
  - 已冻结的大场景序列
  - 已冻结的小场景计划
  - 已完成回合历史
  - 最近一次状态快照
- 如果主线序列存在，但小场景计划缺失，而回合历史里已经出现小场景提交：
  - 应直接判定为恢复状态损坏，而不是静默跳过。

### 4. 幂等策略
- 大场景完成后生成过渡小场景计划时，必须防止重复生成。
- 幂等主键建议：
  - `session_id + from_major_scene_id + to_major_scene_id`
- 如果同一窗口已存在 `active` 或 `completed` 的计划，再次触发时：
  - 直接返回既有计划
  - 不允许重新随机替换

### 5. 事务边界与回滚边界
- `init_training`
  - 事务内完成：
    - 创建训练会话
    - 冻结大场景序列
    - 初始化状态快照
  - 不要求一次性生成所有小场景计划
- `submit_round`
  - 事务内完成：
    - 写入本轮提交
    - 写入评估结果
    - 写入新快照
    - 如本轮完成的是大场景，且需要生成下一个过渡窗口小场景计划，则同事务冻结计划
  - 事务失败则：
    - 本轮回合写入回滚
    - 小场景计划写入回滚
    - 不允许出现“回合成功但计划缺失”的半状态

### 6. 契约维护归属
- 主线和小场景的结构契约由训练后端维护，不交给前端推断。
- 前端或 CLI 只消费：
  - 当前题目
  - 当前题目类型
  - 当前可选项
  - 提交后的状态变化

### 7. 建议的配置结构
- 建议在现有 `scenario_bank.json` 基础上增加 `transition_micro_scene_bank.json`
- 结构建议：
  - `window_id`
  - `from_major_scene_id`
  - `to_major_scene_id`
  - `era_date_range`
  - `candidate_micro_scenes`
- 每个 `candidate_micro_scene` 包含：
  - `id`
  - `title`
  - `brief`
  - `mission`
  - `decision_focus`
  - `options`
  - `target_skills`
  - `risk_tags`
  - `entry_conditions`
  - `forbidden_flags`
  - `state_delta_profile`
  - `selection_weight`

## 风险点
- 如果把小场景完全交给 LLM 动态生成，会直接破坏可恢复性、可回放性和评估一致性。
- 如果小场景只放到前端本地缓存，会造成服务端与前端对“当前会话到哪一步”理解不一致。
- 如果用“提交大场景后再异步补生成小场景”的方式，又不做冻结回写，恢复时很容易出现队列漂移。
- 如果一个大场景之间插入过多小场景，会稀释主线节奏，导致玩家忘记自己处于哪个历史节点。
- 如果小场景也允许任意跳主线，会把训练从“历史情境训练”做成“随机剧情游戏”，偏离项目目标。
- 如果报告层不记录 `scene_level` 与 `parent_major_scene_id`，后续复盘将无法区分“主线错误”和“过程性错误”。

## 测试建议

### 单测
- `transition_scene_plan_policy`
  - 同一历史窗口只会从允许候选池中抽取场景
  - 抽取数量稳定在 `2-3`
  - 禁止命中 `forbidden_flags`
- `scene_timeline_policy`
  - 当存在未完成小场景时，优先返回小场景
  - 当小场景全部完成后，回到下一个主线大场景
- `transition_scene_guard_policy`
  - 禁止跨窗口、跨阶段、跨主线跳转

### 集成测试
- 初始化训练后完成 `S1`，应能冻结 `S1->S2` 过渡窗口的 2-3 个小场景
- 完成小场景后，`get_next_scenario` 应继续返回同一窗口剩余小场景，直到队列清空
- 小场景提交的状态变化应影响后续：
  - 推荐候选
  - 分支触发
  - 报告统计

### 契约测试
- 训练场景 DTO 在主线/小场景/补救支线下均返回稳定结构
- 历史、报告、诊断都能区分：
  - `scene_level`
  - `scene_path_type`
  - `parent_major_scene_id`

### 恢复测试
- 小场景计划已冻结但会话中断后，恢复应读回同一批小场景
- 如果小场景计划丢失但历史已存在对应回合，应返回恢复损坏错误

## 产出要求

### 一、建议的主线与小场景关系
- 主线仍保持 `S1-S6`
- 在以下窗口插入小场景：
  - `W12`: `S1 -> S2`
  - `W23`: `S2 -> S3`
  - `W34`: `S3 -> S4`
  - `W45`: `S4 -> S5`
  - `W56`: `S5 -> S6`

### 二、每个窗口建议准备 4-6 个候选小场景，由后端随机冻结 2-3 个

#### W12: 卢沟桥事变 -> 淞沪会战
- 候选示例：
  - 夜间电报延迟与口径确认
  - 地方广播站快讯二次核验
  - 租界记者来稿冲突比对
  - 民间传闻止损稿审校

#### W23: 淞沪会战 -> 南京失守
- 候选示例：
  - 伤亡名单匿名化处理
  - 撤离路线消息保密判断
  - 外媒线索与本地证词冲突核验
  - 编辑部紧急撤稿说明

#### W34: 南京失守 -> 武汉会战
- 候选示例：
  - 救护站口述采访节选处理
  - 难民安置点位置是否公开
  - 后方宣传稿降温改写
  - 敏感照片说明文字审校

#### W45: 武汉会战 -> 百团大战
- 候选示例：
  - 多站点战报时间戳统一
  - 铁路封锁消息来源复核
  - 后勤物资短缺报道尺度控制
  - 动员海报与新闻稿口径统一

#### W56: 百团大战 -> 终战节点
- 候选示例：
  - 受降传闻的提前发布判断
  - 旧稿纠偏与终战口径合并
  - 街头庆祝引导与秩序提醒
  - 对外电讯标题措辞审校

### 三、推荐的实施顺序
- 第一阶段
  - 先实现一个窗口 `W12` 的 2-3 个小场景可交互闭环
  - 打通冻结、提交、恢复、报告
- 第二阶段
  - 扩到全部五个窗口
  - 把小场景纳入推荐、补救和报告统计
- 第三阶段
  - 在结构不变前提下，再考虑是否为小场景接入受控文案扩写

### 四、明确不做
- 不做运行时完全自由生成的小场景
- 不做让前端自己决定插不插小场景
- 不做小场景直接改写主线大场景历史锚点
- 不做“只加文案不加状态影响”的假交互

### 五、结论
- 这项需求适合做，而且应该做。
- 但正确做法不是简单把 6 个大场景拆碎，而是在主线锚点之间引入“服务端冻结、可恢复、可回放、可影响后续发展”的小场景交互层。
- 训练域最终应呈现为：
  - `6 个历史主线大场景`
  - `5 个主线过渡窗口`
  - `每个窗口随机冻结 2-3 个可交互小场景`
  - `小场景影响状态、分支、推荐和最终报告`
  - `主线锚点仍保持历史叙事稳定`
